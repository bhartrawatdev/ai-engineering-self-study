# ML101 — Classical Machine Learning

## Why This Matters

Classical ML models solve most real business problems more reliably, cheaply, and interpretably than deep learning — and even where a neural network eventually wins, classical models are almost always the right first baseline to build and beat. Without this course, you'd jump straight to deep learning in Volume 4 without ever developing the instinct for when a much simpler model is actually the better engineering choice, or the vocabulary (bias-variance tradeoff, regularization, overfitting) that every later course assumes you already have.

## Prerequisites

- **MA101 (Linear Algebra for ML)** — vectors, matrices, and the PCA/SVD work directly underlie several models and the dimensionality-reduction content here.
- **MA102 (Calculus & Optimization Basics)** — gradient descent from MA102 is the literal training mechanism behind logistic regression and several other models in this course.
- **MA103 (Probability & Statistics for ML)** — model evaluation, overfitting, and the statistical framing of classification/regression all lean directly on MA103's foundations.
- **CS104 (OOP & Software Design)** — this course's Main Project assumes comfort structuring a small multi-model comparison codebase cleanly.

## Where This Is Used Later

- **ML102 (Feature Engineering & Data Pipelines)** — assumes you already know what a model needs from its input data; this course is why that requirement exists in the first place.
- **ML103 (Model Evaluation & Experimentation)** — rigorous evaluation of the models built in this course is exactly ML103's subject; this course provides the models to evaluate.
- **DL101 (Neural Networks From Scratch)** — many core ML concepts (loss functions, overfitting, regularization, train/test splits) carry over directly; DL101 builds on top of this vocabulary rather than re-teaching it.
- **Nearly every AI Engineering course in Volume 6** — knowing when a classical model is the right, simpler tool (instead of always reaching for an LLM) is a genuine engineering judgment call this course builds.

  - The bias-variance tradeoff as the conceptual explanation for why both extremes are bad

  - Random forests — bagging, and why averaging many overfit trees reduces variance

**Topics Covered:**
- Framing classification as predicting a probability, not a raw label
- The sigmoid function — converting a linear combination of features into a value between 0 and 1

**Tools / Technologies Used:**

**Key Techniques / Concepts to Internalize:**

**Practical Tips:**

### Week 3 — Bias-Variance Tradeoff & Regularization

**Topics Covered:**
- Overfitting and underfitting — recognizing each from the gap (or lack of gap) between training and validation performance
- The bias-variance tradeoff: high-bias models (too simple, underfit) vs. high-variance models (too complex, overfit), and why total error is a combination of both
- L2 regularization (Ridge) — penalizing large coefficients, shrinking them toward zero without eliminating them
- L1 regularization (Lasso) — penalizing the absolute value of coefficients, with the practical effect of driving some coefficients exactly to zero (automatic feature selection)
- The regularization strength hyperparameter, and how tuning it moves a model along the bias-variance tradeoff

**Tools / Technologies Used:**
- Python 3.x, scikit-learn, Matplotlib

**Key Techniques / Concepts to Internalize:**
- Overfitting isn't "the model is bad" — it's "the model learned patterns specific to the training data's noise, not the underlying real relationship," which is why it performs well on training data but poorly on new data. This distinction matters because the fix (regularization, more data, a simpler model) is different from fixing a genuinely poorly-specified model.
- The bias-variance tradeoff explains *why* you can't just make a model as complex as possible — complexity reduces bias (the model can represent more relationships) but increases variance (the model becomes more sensitive to the specific training data it saw), and total expected error is the sum of both, plus irreducible noise.
- L1 and L2 regularization both shrink coefficients but do so differently — L2's smooth penalty shrinks everything proportionally, while L1's penalty shape can push coefficients to exactly zero, which is *why* L1 is associated with automatic feature selection and L2 is not.

**Practical Tips:**
- Common mistake: increasing model complexity (e.g., adding polynomial features) to improve training performance without checking validation performance — training error alone can't tell you whether you've started overfitting.
- You understood this week if you can look at a plot of training error and validation error vs. model complexity and correctly identify the point where overfitting begins.
- Deliberately overfit a model on purpose this week (e.g., fit a very high-degree polynomial to a small, noisy dataset) and watch it happen — seeing the wild, noise-chasing curve directly is far more convincing than reading the definition of overfitting.

**Recommended Resources:**

- 📘 Book: *Hands-On Machine Learning* — Chapter 4 (Regularized Linear Models section)
- 📄 Docs: [scikit-learn — Ridge and Lasso](https://scikit-learn.org/stable/modules/linear_model.html#ridge-regression-and-classification)
- 🤖 AI Prompt: "Give me a small, noisy synthetic dataset. ### Week 4 — Decision Trees & Ensemble Methods

**Topics Covered:**
- Decision trees: how splits are chosen at each node (Gini impurity or entropy, at a conceptual level — not deriving the formulas by hand), tree depth and overfitting
- Why a single unconstrained decision tree tends to overfit badly (low bias, very high variance)
- Random forests: bagging (bootstrap aggregating) — training many trees on random subsets of data and features, then averaging their predictions
- Why averaging many high-variance, low-bias trees produces a lower-variance ensemble without sacrificing much bias
- Feature importance from tree-based models — a genuinely useful, interpretable byproduct

**Tools / Technologies Used:**
- Python 3.x, scikit-learn (`DecisionTreeClassifier`, `RandomForestClassifier`), Matplotlib

**Key Techniques / Concepts to Internalize:**
- A decision tree is, at its core, a sequence of "if this feature is above/below some threshold, go left/right" questions — this makes trees naturally interpretable (you can literally read the decision path), which is a genuine practical advantage over many other model types, including neural networks.
- Random forests directly demonstrate the bias-variance tradeoff from Week 3 in action: each individual tree has low bias but high variance, and averaging many such trees (bagging) reduces the variance substantially while barely increasing bias — this is a concrete, memorable example of managing the tradeoff through ensembling rather than through regularization alone.
- Feature importance scores from a random forest tell you which features the model relied on most, but they're not the same as causal importance — a feature can be "important" to the model's predictions due to correlation with the real cause, echoing MA103's correlation-vs-causation caution.

**Practical Tips:**
- Common mistake: leaving a decision tree completely unconstrained (no `max_depth` or other stopping criteria) and being surprised when it overfits dramatically — always set at least a reasonable depth limit or minimum samples per leaf as a starting point.
- You understood this week if you can explain, without notes, why a random forest tends to generalize better than any single one of its constituent trees.
- Visualize at least one small decision tree directly (scikit-learn can render the tree structure) — seeing the actual splits and thresholds it chose makes the abstract "how trees make decisions" concept fully concrete.

### Week 5 — Gradient Boosting & K-Nearest Neighbors

**Topics Covered:**
- Gradient boosting, conceptually: building trees sequentially, where each new tree is trained to correct the errors (residuals) of the ensemble so far, rather than all trees being trained independently as in bagging
- Practical use of a modern gradient boosting library (XGBoost or LightGBM) on a tabular dataset
- Why gradient-boosted trees are frequently the strongest performer on structured/tabular data specifically, even compared to deep learning
- K-Nearest Neighbors (KNN): a non-parametric approach — predicting based on the k most similar training examples, with no explicit "training" phase beyond storing the data
- The curse of dimensionality's practical effect on KNN — why distance-based methods degrade as feature count grows

**Tools / Technologies Used:**
- Python 3.x, scikit-learn, XGBoost or LightGBM

**Key Techniques / Concepts to Internalize:**
- Bagging (Week 4) and boosting (this week) both combine many trees but for opposite reasons — bagging reduces variance by averaging independent, high-variance trees; boosting reduces bias by sequentially, deliberately targeting the current ensemble's mistakes. Confusing the two mechanisms is common, and this contrast is worth holding clearly in mind.
- Gradient boosting's sequential nature means it can overfit if allowed to run for too many rounds (each new tree is fitting ever-smaller residuals, eventually chasing noise) — this is why boosting libraries expose early stopping and learning rate hyperparameters as first-class, important controls, not optional extras.
- KNN's "curse of dimensionality" is a genuinely important, broadly applicable idea beyond just this algorithm: in high-dimensional spaces, the notion of "nearby" becomes less meaningful because most points end up roughly equidistant from each other — this insight resurfaces conceptually in later courses whenever distance-based reasoning is used on high-dimensional data (e.g., embeddings in NLP).

**Practical Tips:**
- Common mistake: training a gradient boosting model for far too many rounds without early stopping, letting it overfit to training-set noise — always monitor validation performance during training and stop when it plateaus or worsens.
- You understood this week if you can explain, in one or two sentences, the core mechanistic difference between how bagging and boosting build their ensembles.
- Compare KNN's performance directly on a low-dimensional dataset vs. an artificially high-dimensional version of the same data (e.g., adding many random noise features) — watching its accuracy degrade concretely demonstrates the curse of dimensionality rather than just describing it.

**Recommended Resources:**

- 📘 Book: *Hands-On Machine Learning* — Chapter 7 (Boosting section)
- 📄 Docs: [XGBoost — Python API documentation](https://xgboost.readthedocs.io/en/stable/python/python_intro.html), [scikit-learn — Nearest Neighbors](https://scikit-learn.org/stable/modules/neighbors.html)

Have me fit a gradient boosting model (XGBoost or LightGBM) with early stopping based on a validation set, compare it to the random forest from last week, and have me explain in my own words why boosting and bagging arrived at their results through different mechanisms."

### Week 6 — Support Vector Machines

**Topics Covered:**
- The margin — the core idea behind SVMs: finding the decision boundary that maximizes the distance to the nearest points of each class
- Support vectors — the specific data points that actually determine the boundary, and why most training points don't matter to the final model
- The kernel trick, conceptually: implicitly mapping data into a higher-dimensional space to make a non-linearly-separable problem linearly separable, without explicitly computing that transformation
- Common kernels (linear, RBF) and when each is appropriate
- Where SVMs still fit in a modern toolkit vs. where gradient-boosted trees or other methods have largely superseded them

**Tools / Technologies Used:**
- Python 3.x, scikit-learn (`SVC`)

**Key Techniques / Concepts to Internalize:**
- Maximizing the margin (not just finding *any* separating boundary) is what gives SVMs strong generalization properties — a boundary that barely separates the classes is more sensitive to new data than one with maximum breathing room on both sides, which connects back to the bias-variance tradeoff.
- The kernel trick is a genuinely clever piece of math worth understanding conceptually even without deriving it: it lets you compute what the result *would be* if you'd transformed your data into a much higher-dimensional space, without ever actually performing that expensive transformation directly.
- SVMs, once dominant for many classification tasks, are today most commonly outperformed by gradient-boosted trees on tabular data and by deep learning on unstructured data (images, text) — understanding *why* this shift happened (boosting's flexibility and scalability, deep learning's representation learning) is more valuable than mastering SVM tuning in depth.

**Practical Tips:**
- Common mistake: applying an SVM with an RBF kernel to a large dataset without considering training time — SVMs scale poorly to large numbers of training examples compared to tree-based methods, which is a real practical constraint, not just a theoretical one.
- You understood this week if you can explain, without notes, what a support vector actually is and why most training points aren't one.
- Visualize an SVM's decision boundary and margin on a simple 2D dataset with both a linear and an RBF kernel, side by side, to see concretely what the kernel trick enables.

- 📘 Book: *Hands-On Machine Learning* — Chapter 5 (Support Vector Machines)

### Week 7 — Unsupervised Learning: Clustering & Dimensionality Reduction

**Topics Covered:**
- K-Means clustering: the algorithm (initialize centroids, assign points, update centroids, repeat), and its core assumption (roughly spherical, similarly-sized clusters)
- Choosing k: the elbow method and silhouette score as practical (imperfect) tools
- Limitations of K-Means: sensitivity to initialization, poor fit for non-spherical or unevenly-sized clusters
- Revisiting PCA (from MA101) explicitly in an ML workflow context: as a preprocessing step before clustering/modeling, and as a visualization tool for high-dimensional data
- The general distinction this week reinforces: supervised learning needs labels, unsupervised learning finds structure without them

**Tools / Technologies Used:**
- Python 3.x, scikit-learn (`KMeans`, `PCA`), Matplotlib

**Key Techniques / Concepts to Internalize:**
- K-Means's assumption of roughly spherical clusters isn't a minor technical detail — it's the reason K-Means can fail badly on real data with elongated, irregularly-shaped, or very differently-sized clusters, even when a human looking at a 2D plot could easily see the "real" groups. Recognizing this limitation is more useful than just knowing the algorithm's steps.
- Choosing k is fundamentally somewhat subjective — the elbow method and silhouette score are useful heuristics, not definitive answers, and different reasonable choices of k can both be "correct" depending on what granularity of grouping is actually useful for the problem at hand.
- Using PCA before clustering (or before feeding data into other models) isn't just about visualization — reducing dimensionality can also reduce noise and mitigate the curse of dimensionality's effect on distance-based methods like K-Means, directly connecting back to Week 5's KNN discussion.

**Practical Tips:**
- Common mistake: running K-Means on data with very different feature scales without normalizing first — since K-Means relies on distance, a feature with a much larger numeric range will dominate the clustering regardless of its actual importance.
- You understood this week if you can look at a 2D scatter plot of clearly non-spherical clusters (e.g., two interleaving moon shapes) and correctly predict that K-Means will struggle with it before running the algorithm.
- Try K-Means with several different values of k and several different random initializations on the same dataset, and observe how much the resulting clusters can vary — this concretely demonstrates why K-Means's sensitivity to initialization is a real practical concern, not a theoretical footnote.

### Week 8 — Model Selection & Wrap-Up

**Topics Covered:**
- Cross-validation: why a single train/test split can give a misleadingly optimistic or pessimistic estimate of performance, and how k-fold cross-validation addresses this
- Hyperparameter tuning: grid search and random search, using scikit-learn's `GridSearchCV`/`RandomizedSearchCV`
- A practical model-selection framework: balancing predictive performance, interpretability, training/inference cost, and data size/type against each other, rather than always defaulting to "the most accurate model"
- Course integration: bringing together everything from Weeks 1–7 into the Main Project's model comparison

**Tools / Technologies Used:**
- Python 3.x, scikit-learn (`cross_val_score`, `GridSearchCV`, `RandomizedSearchCV`)

**Key Techniques / Concepts to Internalize:**
- A single train/test split's performance estimate depends partly on which specific rows happened to land in the test set — k-fold cross-validation averages performance across multiple different splits, giving a more reliable, less "lucky or unlucky split"-dependent estimate.
- Hyperparameter tuning via grid/random search is applying the exact same "compare candidates on held-out data" principle as basic train/test evaluation, just automated across many configurations — recognizing it as the same underlying idea, scaled up, prevents it from feeling like a separate, unrelated technique.
- The "best" model isn't always the one with the highest raw accuracy or lowest error — a slightly less accurate but much more interpretable model (like a single decision tree over a large ensemble) is sometimes the genuinely correct engineering choice, especially when stakeholders need to understand *why* a prediction was made.

**Practical Tips:**
- Common mistake: tuning hyperparameters using the test set directly (rather than cross-validation on the training set, with the test set held out entirely until final evaluation) — this leaks test set information into model selection and produces an overly optimistic final performance estimate.
- You understood this week if you can explain why k-fold cross-validation gives a more trustworthy performance estimate than a single train/test split, and set up a `GridSearchCV` correctly without letting the test set leak into the process.
- Treat this week as integration, not new isolated material — the goal is running a genuinely rigorous model comparison across everything from Weeks 1–7 as the Main Project, not learning cross-validation and hyperparameter tuning in isolation.
