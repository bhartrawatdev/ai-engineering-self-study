# CS105 — Databases & SQL Basics

## Why This Matters

Almost every real AI system sits on top of structured data somewhere — training data, experiment logs, user data, or the metadata behind a RAG pipeline's vector store. SQL is also one of the most durable, broadly transferable skills in this entire program: virtually every data-adjacent job, ML or not, assumes you can query and reason about relational data. This course closes out Volume 2 by making sure "where does the data actually live and how do I get it out efficiently" is never a blocker in later, more AI-specific courses.

## Prerequisites

- **CS101 (Programming Foundations)** — comfortable with Python, since this course connects SQL to real Python code (via `sqlite3` and a basic ORM) rather than treating SQL in isolation.
- **CS104 (OOP & Software Design)** — the ORM section of this course (SQLAlchemy) leans on class-based design; recommended, though CS105 can be worked through with only CS101 if needed.

## Where This Is Used Later

- **ML102 (Feature Engineering & Data Pipelines)** — real data pipelines very often pull from or write to relational databases; this course's SQL fluency is a direct prerequisite.
- **ML103 (Model Evaluation & Experimentation)** — this course's Main Project (an experiment tracker) previews exactly the kind of structured logging ML103's evaluation workflows depend on.
- **AIE101 (Retrieval-Augmented Generation Systems)** — vector databases (introduced conceptually in Week 6 here) are the backbone of RAG systems; this course is the deliberate bridge into that topic.
- **AIE104 (MLOps & Model Deployment)** — production systems almost always have a database somewhere in the stack (model registries, feature stores, logging); this course is the foundation for reasoning about that layer.

## Full Topic Breakdown

- **The Relational Model & Basic SQL**
  - Tables, rows, columns, primary keys — the relational model's core vocabulary
  - `SELECT`, `WHERE`, `ORDER BY`, `LIMIT` — retrieving and filtering data
  - Data types in SQL, and `NULL` — what it means and how it behaves differently from other values
- **Joins & Aggregations**
  - Foreign keys — how tables relate to each other
  - `INNER JOIN`, `LEFT JOIN` (and briefly `RIGHT`/`FULL JOIN`) — combining data across tables
  - Aggregate functions: `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`
  - `GROUP BY` and `HAVING` — aggregating and filtering aggregated results
- **Schema Design & Normalization**
  - Designing a schema from a real-world scenario — entities, relationships, keys
  - Normalization: 1NF, 2NF, 3NF — conceptual understanding of what each level prevents (specifically data redundancy and update anomalies)
  - When to deliberately denormalize, and the tradeoffs involved
- **Indexes, Query Performance & Transactions**
  - What an index actually is (conceptually connects back to CS103's trees/hash tables), and why it speeds up lookups
  - When an index helps vs. when it doesn't (or actively hurts write performance)
  - Transactions: ACID properties, `COMMIT`/`ROLLBACK`, why transactions matter for data integrity
- **Connecting Python to Databases**
  - Using `sqlite3` directly: connections, cursors, executing queries, avoiding SQL injection with parameterized queries
  - Introduction to SQLAlchemy as an ORM: mapping Python classes to tables, basic querying through the ORM
  - When to use raw SQL vs. an ORM, and why both skills matter
- **Beyond Relational: NoSQL & Vector Database Concepts**
  - Why NoSQL databases exist — the specific problems they solve that relational databases handle less naturally (unstructured/semi-structured data, horizontal scaling)
  - Document stores (conceptual, e.g. MongoDB-style) vs. key-value stores vs. relational — a comparison, not deep implementation
  - Vector databases — what they store (embeddings), why similarity search is fundamentally different from exact-match SQL queries, and why this matters directly for AIE101's RAG systems

### Week 1 — The Relational Model & Basic SQL

**Topics Covered:**
- Tables, rows, columns, and primary keys — the vocabulary of the relational model
- `SELECT` statements: choosing columns, `WHERE` for filtering, `ORDER BY` for sorting, `LIMIT` for restricting result size
- Comparison and logical operators in `WHERE` clauses (`=`, `!=`, `<`, `>`, `AND`, `OR`, `IN`, `BETWEEN`, `LIKE`)
- `NULL` — what it represents (absence of a value, not zero or empty string), and why `= NULL` doesn't work the way you'd expect (`IS NULL` instead)

**Tools / Technologies Used:**
- SQLite (via the `sqlite3` command-line tool or a GUI like DB Browser for SQLite), SQL

**Key Techniques / Concepts to Internalize:**
- A relational database models the world as tables of consistently-shaped rows — this rigidity is a feature, not a limitation, because it's exactly what makes querying, joining, and reasoning about the data reliable and predictable.
- A primary key's job is guaranteeing every row is uniquely identifiable — this single idea underlies everything about how tables relate to each other later (foreign keys, Week 2).
- `NULL` behaves unlike any other value in SQL — comparisons involving `NULL` (like `column = NULL`) evaluate to unknown, not true or false, which is why `IS NULL`/`IS NOT NULL` exist as special-case operators. This trips up nearly everyone learning SQL at least once.

**Practical Tips:**
- Common mistake: writing `WHERE column = NULL` expecting it to filter for null values — it silently returns no rows instead of erroring, which makes the bug easy to miss.
- You understood this week if you can write a `SELECT` query with multiple `WHERE` conditions, sorting, and a row limit, entirely from memory, for a table you've never seen the schema of before (just given column names).
- Practice on a real, moderately-sized dataset (not a 5-row toy table) — SQLite makes it trivial to load a CSV of a few thousand rows and query it, and small toy tables hide bugs that only show up with more realistic data volumes.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "SQL Tutorial - Full Database Course for Beginners" (first ~1 hour, through basic SELECT/WHERE)
- 📘 Book: *SQL Queries for Mere Mortals* by John L. Viescas — Chapters 1–4
- 📄 Docs: [sqlite.org/lang_select.html](https://www.sqlite.org/lang_select.html)
- 🤖 AI Prompt: "Give me a small sample table schema (like a 'products' table with name, price, category, stock). Have me write 5 SELECT queries of increasing complexity to answer specific questions about the data, then check my SQL syntax and logic."

### Week 2 — Joins & Aggregations

**Topics Covered:**
- Foreign keys — how one table references another, and what "referential integrity" means
- `INNER JOIN` — combining rows from two tables where a match exists
- `LEFT JOIN` — combining rows and keeping all rows from the left table even without a match (and briefly, `RIGHT JOIN`/`FULL JOIN` as the same idea from the other direction/combined)
- Aggregate functions: `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`
- `GROUP BY` — aggregating data by category, and `HAVING` — filtering on the result of an aggregation (distinct from `WHERE`, which filters before aggregation)

**Tools / Technologies Used:**
- SQLite, SQL

**Key Techniques / Concepts to Internalize:**
- A JOIN doesn't "merge" tables into one physical thing — it computes a new result set on the fly by matching rows based on a condition (usually a foreign key = primary key relationship), and understanding this as a computation, not a storage change, clarifies a lot of confusion about how joins actually work.
- `INNER JOIN` vs. `LEFT JOIN` is really a question about what to do with non-matches — `INNER JOIN` drops them, `LEFT JOIN` keeps the left table's row with `NULL`s filling in for the missing right-side columns. Choosing wrong (using `INNER JOIN` when you needed `LEFT JOIN`) is a very common source of silently missing data in query results.
- `WHERE` filters rows *before* grouping; `HAVING` filters groups *after* aggregation — this ordering distinction is exactly why you can't use an aggregate function like `COUNT(*)` inside a `WHERE` clause but can inside `HAVING`.

**Practical Tips:**
- Common mistake: using `INNER JOIN` when some rows in the left table genuinely have no match in the right table, silently losing those rows from the result without an error — always ask "could there be unmatched rows here, and do I want to keep them?"
- You understood this week if you can look at a two-table schema (e.g. `orders` and `customers`) and correctly write a query joining them with an aggregate (e.g. "total order value per customer") without hesitation.
- Build the habit of predicting a join's row count before running it (roughly) — if your predicted count is wildly different from the actual result, that's often a sign of an accidental many-to-many join blowing up your row count unexpectedly.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "SQL Tutorial" continued (Joins, Aggregate Functions, GROUP BY sections)
- 📘 Book: *SQL Queries for Mere Mortals* — Chapters 6–8 (Joins), Chapter 9 (Grouping)
- 📄 Docs: [sqlite.org/lang_select.html#joins](https://www.sqlite.org/lang_select.html)
- 🤖 AI Prompt: "Give me two related table schemas (like 'orders' and 'customers', connected by customer_id). Have me write 4-5 queries involving joins and aggregations of increasing complexity, then check each one and point out if I chose the wrong join type for the intended result."

**This Week's Mini Exercise:**
Design two related tables (e.g. `customers` and `orders`), populate each with a handful of rows including at least one customer with no orders, then write queries using `INNER JOIN` and `LEFT JOIN` on the same tables and explicitly compare the different row counts/results each produces — explaining in a comment why they differ.

### Week 3 — Schema Design & Normalization

**Topics Covered:**
- Designing a schema from a real-world scenario: identifying entities, their attributes, and the relationships between them
- One-to-many, many-to-many relationships, and junction/bridge tables for representing many-to-many
- Normalization: First Normal Form (1NF), Second Normal Form (2NF), Third Normal Form (3NF) — what each specifically prevents (repeating groups, partial dependency, transitive dependency)
- Update, insertion, and deletion anomalies — the concrete problems normalization exists to solve
- When and why to deliberately denormalize (performance tradeoffs), even though this course teaches normalized design as the default

**Tools / Technologies Used:**
- SQLite, SQL, pen-and-paper (or a simple diagramming tool) for schema design

**Key Techniques / Concepts to Internalize:**
- Normalization isn't an abstract academic exercise — each normal form directly prevents a specific, concrete bug: storing the same information redundantly in multiple rows (so an update in one place doesn't match another), or being unable to represent certain facts without also representing unrelated ones. Learning normalization through "what breaks if I don't do this" is far stickier than memorizing the formal definitions.
- A many-to-many relationship (e.g. students and courses, where each student takes many courses and each course has many students) can't be represented with a simple foreign key on either table — it needs a junction table with foreign keys to both sides. Recognizing this pattern is one of the most common real schema design decisions.
- Denormalization is a legitimate, deliberate tradeoff (trading some redundancy for query performance), not a mistake — but it should be a conscious choice made after understanding the normalized version, not a default reached out of not knowing how to normalize in the first place.

**Practical Tips:**
- Common mistake: designing a schema with repeating groups (e.g. `phone1`, `phone2`, `phone3` columns instead of a separate `phone_numbers` table) — if you're naming columns with numbers appended, that's usually a sign a separate related table is needed instead.
- You understood this week if, given an unnormalized spreadsheet-style dataset (with obvious redundancy), you can redesign it into a properly normalized multi-table schema and explain which anomaly each change prevents.
- Sketch schemas on paper (or a simple ER diagram tool) before writing any `CREATE TABLE` statements — designing directly in SQL syntax tends to lock in a first-draft structure before you've actually thought through the relationships.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "SQL Tutorial" continued (Database design and normalization sections), or search "Database normalization explained" for a focused explainer
- 📘 Book: *SQL Queries for Mere Mortals* — Chapters 2–3 (Design theory) and any normalization-specific chapter
- 📄 Docs: N/A — best learned through a hands-on redesign exercise; use the mini exercise and AI prompt instead
- 🤖 AI Prompt: "Give me a single, deliberately unnormalized table representing something like 'student course enrollments' with repeating groups and redundant data. Have me redesign it into a properly normalized 3NF multi-table schema, then check my design and point out any remaining anomalies."

**This Week's Mini Exercise:**
Take a deliberately unnormalized, single-table dataset (e.g., an "orders" table that repeats customer name/address on every row, and has `item1`, `item2`, `item3` columns for multiple items per order), redesign it into a properly normalized multi-table schema (likely 3+ tables with a junction table for the many-to-many order-items relationship), write the `CREATE TABLE` statements, and migrate the sample data into the new structure.

### Week 4 — Indexes, Query Performance & Transactions

**Topics Covered:**
- What an index actually is: a separate, ordered structure (often a B-tree — connecting directly back to CS103's trees) that lets the database find rows without scanning the whole table
- When an index helps (columns frequently used in `WHERE`/`JOIN`/`ORDER BY`) vs. when it doesn't help or actively hurts (small tables, columns rarely queried, and the write-performance cost of maintaining an index)
- Reading a basic query execution plan (`EXPLAIN QUERY PLAN` in SQLite) to see whether an index is actually being used
- Transactions: what "ACID" (Atomicity, Consistency, Isolation, Durability) means in practice, `BEGIN`/`COMMIT`/`ROLLBACK`
- Why transactions matter for data integrity — a concrete scenario where a multi-step operation failing partway through would corrupt data without one

**Tools / Technologies Used:**
- SQLite, SQL

**Key Techniques / Concepts to Internalize:**
- An index trades write speed and storage space for read speed — every index you add makes `INSERT`/`UPDATE`/`DELETE` slightly slower (the index has to be maintained too), so indexing every column "just in case" is a real anti-pattern, not free performance.
- Indexes directly connect back to CS103: a typical B-tree index gives you the same O(log n) search behavior as the BSTs from that course, applied to database rows instead of in-memory objects — recognizing this isn't a coincidence but the same underlying idea reapplied.
- A transaction's "atomicity" guarantee — a multi-step operation either fully completes or fully doesn't happen — is what prevents a genuinely common real bug: a system crashing or erroring midway through a multi-table update, leaving the data in a half-changed, inconsistent state.

**Practical Tips:**
- Common mistake: assuming more indexes always mean a faster database — on a small table, or on a column rarely used for filtering, an index is pure overhead with no benefit.
- You understood this week if you can look at a slow query, hypothesize which column(s) would benefit from an index, add it, and confirm via `EXPLAIN QUERY PLAN` that the index is actually being used.
- Deliberately create a scenario without a transaction where a simulated failure midway through a multi-step operation leaves inconsistent data, then fix it by wrapping the operation in a transaction — experiencing the failure mode directly makes the guarantee feel necessary rather than theoretical.

**Recommended Resources:**
- 📺 YouTube: search "SQLite indexes explained" or "Database indexing explained" for a focused, visual explainer (channel varies by current availability)
- 📘 Book: *SQL Queries for Mere Mortals* — performance/indexing sections if covered; supplement with SQLite's own docs, which are unusually readable
- 📄 Docs: [sqlite.org/queryplanner.html](https://www.sqlite.org/queryplanner.html), [sqlite.org/lang_transaction.html](https://www.sqlite.org/lang_transaction.html)
- 🤖 AI Prompt: "Give me a moderately large sample table (describe it, no need for real data) and a slow-sounding query on it (e.g. filtering on a non-indexed column). Have me reason through whether adding an index would help, which column(s) I'd index, and what the tradeoff is, before confirming my reasoning."

**This Week's Mini Exercise:**
Using a table with at least a few thousand rows (generate synthetic data if needed), run a query filtering on a non-indexed column and note the time/query plan, then add an index on that column, re-run the same query, and compare the execution plan and timing before and after — write a short note on what changed and why.

### Week 5 — Connecting Python to Databases

**Topics Covered:**
- Using Python's built-in `sqlite3` module: opening a connection, creating a cursor, executing queries, fetching results
- Parameterized queries — passing values safely into a query instead of string-formatting them directly, and why this specifically prevents SQL injection
- Introduction to SQLAlchemy as an ORM: defining Python classes mapped to database tables, basic querying through the ORM instead of raw SQL strings
- When raw SQL is preferable (complex queries, performance-critical paths) vs. when an ORM's convenience wins (typical CRUD operations, working within a larger typed Python codebase)

**Tools / Technologies Used:**
- Python 3.x, `sqlite3` (standard library), SQLAlchemy

**Key Techniques / Concepts to Internalize:**
- SQL injection isn't an exotic attack — it happens the moment you build a query string by directly concatenating user input (`f"SELECT * FROM users WHERE name = '{name}'"`), and parameterized queries (`cursor.execute("SELECT * FROM users WHERE name = ?", (name,))`) close this gap by treating input strictly as data, never as executable SQL. This should become an unbreakable habit starting this week, not an occasional precaution.
- An ORM maps database rows to Python objects, letting you work with familiar classes and attributes instead of raw SQL strings — this connects directly back to CS104's OOP work, and recognizing an ORM as "just" a structured translation layer (not magic) makes debugging it much less mysterious.
- Neither raw SQL nor an ORM is universally "better" — an ORM's abstraction can hide what's actually happening (sometimes generating inefficient queries under the hood), while raw SQL gives full control at the cost of more manual work; real projects use both, choosing per situation.

**Practical Tips:**
- Common mistake: building SQL query strings with f-strings or `.format()` using untrusted input — even in a personal project with no real "attacker," building this habit correctly now avoids a serious, easy-to-introduce vulnerability in any future real system.
- You understood this week if you can write the same basic CRUD (Create, Read, Update, Delete) operations both using raw parameterized `sqlite3` queries and using SQLAlchemy's ORM, and explain what's different about the two approaches.
- Inspect the actual SQL an ORM generates at least once (SQLAlchemy can log the SQL it produces) — seeing the translation from Python object operations to real SQL underneath demystifies the abstraction.

**Recommended Resources:**
- 📺 YouTube: Corey Schafer — "Python SQLite Tutorial" and freeCodeCamp/similar — "SQLAlchemy ORM Tutorial"
- 📘 Book: *SQLAlchemy* documentation's own tutorial is genuinely good enough to serve as the primary text here
- 📄 Docs: [docs.python.org/3/library/sqlite3.html](https://docs.python.org/3/library/sqlite3.html), [docs.sqlalchemy.org/en/20/orm/quickstart.html](https://docs.sqlalchemy.org/en/20/orm/quickstart.html)
- 🤖 AI Prompt: "I have a simple 'users' table with id, name, and email. Walk me through implementing create, read, update, and delete operations first using raw sqlite3 with parameterized queries, then using SQLAlchemy's ORM for the same operations — have me write each version and check it for correctness and safety."

**This Week's Mini Exercise:**
Build a small Python CLI tool (using `argparse` from CS101) that performs CRUD operations on a SQLite `tasks` table (id, description, done) — implement it once using raw `sqlite3` with parameterized queries, then reimplement the same functionality using SQLAlchemy's ORM, and confirm both versions produce identical results.

### Week 6 — Beyond Relational: NoSQL & Vector Database Concepts

**Topics Covered:**
- Why NoSQL databases exist: specific problems relational databases handle less naturally — flexible/evolving schemas, unstructured or semi-structured data, horizontal scaling across many machines
- Document stores (conceptually, e.g. MongoDB-style JSON documents) vs. key-value stores vs. relational — comparing the mental models, not implementing each
- Vector databases: what they actually store (high-dimensional embedding vectors, directly connecting back to MA101's vector spaces), and why "similarity search" (nearest neighbor in vector space) is a fundamentally different query pattern than SQL's exact-match/range filtering
- Why this matters directly for AIE101 (RAG systems): a RAG pipeline's retrieval step is, at its core, a vector database similarity search

**Tools / Technologies Used:**
- Python 3.x, a lightweight local vector search library for the exercise (e.g. `numpy` for a from-scratch nearest-neighbor search, or a simple library like `chromadb` if you want a preview of a real vector database's API)

**Key Techniques / Concepts to Internalize:**
- NoSQL isn't "the modern replacement for SQL" — it's a different set of tradeoffs (often trading strict consistency/schema rigidity for flexibility and horizontal scale), appropriate for specific problems, not a strictly better default. A well-designed relational schema is still the right tool for most structured, relationship-heavy data.
- A vector database's core operation — "find the k most similar vectors to this query vector" — is conceptually identical to a nearest-neighbor search over points in space (MA101's vector norms are literally the distance metric being used), just at a scale and dimensionality where naive approaches need specialized indexing structures (approximate nearest neighbor algorithms) to stay fast.
- This week is intentionally a conceptual bridge, not a full NoSQL/vector-database implementation course — the goal is recognizing *when* each kind of database fits a problem, and specifically understanding vector search well enough that AIE101's RAG systems don't feel like an unfamiliar new paradigm.

**Practical Tips:**
- Common mistake: assuming a vector database replaces the need for a relational database in a RAG system — in practice, real RAG systems typically use both: a vector store for semantic similarity search, and a relational (or document) store for the actual structured metadata/content.
- You understood this week if you can explain, in plain language, why you can't just use a SQL `WHERE` clause to find "documents similar in meaning to this one" — and what a vector embedding plus distance metric does instead.
- Implement a tiny nearest-neighbor search completely from scratch with NumPy (a handful of vectors, brute-force distance calculation) before looking at any real vector database library — the goal is understanding the underlying operation, not learning a specific tool's API yet.

**Recommended Resources:**
- 📺 YouTube: search "vector databases explained" for a current, focused conceptual explainer (the field moves quickly, so prefer something recent)
- 📘 Book: N/A — this is a fast-moving, practically-oriented topic better covered by current documentation/blog content than a book
- 📄 Docs: [MongoDB — NoSQL vs SQL comparison](https://www.mongodb.com/nosql-explained/nosql-vs-sql) (read for the conceptual comparison, not as a MongoDB tutorial), [Pinecone — "What is a Vector Database?"](https://www.pinecone.io/learn/vector-database/) (a widely-used, clear conceptual explainer)
- 🤖 AI Prompt: "Explain the difference between how a SQL WHERE clause finds matching rows and how a vector database finds 'similar' items, using a concrete example like searching for similar product descriptions. Then have me sketch (in words) what a brute-force nearest-neighbor search over embeddings would look like before showing me how real vector databases optimize this at scale."

## Main Project

**Project: ML Experiment Tracker (Database-Backed)**

A small database-backed CLI tool for logging and querying machine learning experiment runs — hyperparameters, metrics, and metadata — designed as a properly normalized relational schema with both raw SQL and ORM access paths. This project exists to be genuinely useful going forward: you'll have real, hands-on use for exactly this kind of tracking from ML101 onward.

**Requirements:**
- Design a normalized schema (at least 3 tables) representing: experiments/runs, hyperparameters (allowing an arbitrary, varying set of hyperparameter names/values per run — think about how to model this without repeating-group columns), and metrics (allowing multiple metrics logged per run, potentially at different steps/epochs).
- Implement a Python CLI tool (using `argparse`) supporting at minimum: logging a new run with its hyperparameters, logging a metric value for an existing run, and querying/listing runs with filtering (e.g., "show me all runs with a validation accuracy above X").
- Implement the core CRUD operations using raw parameterized `sqlite3` SQL.
- Implement the same core operations a second way, using SQLAlchemy's ORM, as a separate module — proving you can move between both approaches.
- Include at least one deliberately indexed column (based on what you'd realistically filter/sort runs by often) and confirm via `EXPLAIN QUERY PLAN` that it's used.
- Wrap any multi-step write operation (e.g., logging a run plus all its hyperparameters at once) in a transaction.

**Stretch Goals:**
- Add a simple "compare runs" command that shows two or more runs' hyperparameters and metrics side by side.
- Export query results (e.g., "all runs sorted by best validation metric") to a CSV using what you learned in CS101.
- Add a basic vector-similarity stretch: store a fake "run embedding" (a random vector standing in for some future real similarity notion) per run, and implement a "find similar runs" command using the brute-force nearest-neighbor approach from Week 6.

**What "Done" Looks Like:**
- The schema is properly normalized — no repeating hyperparameter/metric columns, and you can explain which normal form violations your design specifically avoids.
- Both the raw-SQL and SQLAlchemy-ORM code paths work correctly and produce identical results for the same operations.
- You can log a handful of realistic-looking experiment runs and successfully query/filter/compare them through the CLI.
- Pushed to this repo's `Projects/` folder with a README explaining the schema design and how to use the CLI.
