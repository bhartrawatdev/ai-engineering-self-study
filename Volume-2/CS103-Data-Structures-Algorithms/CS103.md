# CS103 — Data Structures & Algorithms

## Why This Matters

CS101 taught you Python's built-in structures (lists, dicts, sets); this course teaches you *why* they behave the way they do, what's underneath them, and when to reach for something else entirely. Every time you deal with a large dataset, a slow training loop, or a system that needs to scale, the difference between "it works" and "it works fast enough" is almost always a data structure or algorithm choice. This is also the course that builds the kind of rigorous, complexity-aware thinking that shows up constantly in ML engineering — from choosing efficient data pipelines to understanding why certain model operations are expensive.

## Prerequisites

- **CS101 (Programming Foundations)** — comfortable with Python's built-in data structures, functions, recursion, and classes.
- **CS102 (Git, Linux & Dev Tooling)** — this course's project work assumes a normal Git workflow without needing to think about it.

## Where This Is Used Later

- **CS104 (OOP & Software Design)** — many of this course's custom data structures are natural, concrete examples for practicing real class design.
- **ML102 (Feature Engineering & Data Pipelines)** — efficient data structures and complexity-aware thinking directly affect how well a data pipeline scales.
- **DL101 (Neural Networks From Scratch)** — computational graphs (used for backprop) are literally graph data structures; graph traversal concepts from this course apply directly.
- **AIE102 (AI Agents & Tool Use)** and **AIE105 (Scalable AI Systems)** — reasoning about algorithmic complexity and efficient data structures underlies almost all real production AI engineering work, not just "traditional" software engineering.

## Full Topic Breakdown

- **Complexity Analysis**
  - Big O, Big Θ, Big Ω — what each actually describes
  - Time complexity vs. space complexity
  - Analyzing loops, nested loops, and recursive calls for complexity
  - Common complexity classes: O(1), O(log n), O(n), O(n log n), O(n²), O(2ⁿ) — and recognizing which real operations fall into each
- **Linear Data Structures**
  - Arrays vs. linked lists — tradeoffs in access, insertion, deletion
  - Singly and doubly linked lists, implementing from scratch
  - Stacks — LIFO, use cases (undo systems, expression evaluation, call stacks)
  - Queues — FIFO, use cases (task scheduling, BFS)
- **Recursion & Backtracking**
  - Recursion revisited with complexity analysis (recursion trees, recurrence relations at an intuitive level)
  - Backtracking as a pattern: try, recurse, undo if it doesn't work
  - Classic backtracking problems (permutations, combinations, simple constraint satisfaction)
- **Trees**
  - General tree terminology (root, leaf, depth, height)
  - Binary trees, binary search trees (BST) — insertion, deletion, search
  - Tree traversal: in-order, pre-order, post-order, level-order
  - Balanced vs. unbalanced trees (conceptual introduction — not full self-balancing tree implementation)
- **Hash Tables & Heaps**
  - How a hash table actually works: hashing, buckets, collision handling
  - Why average-case O(1) lookup is possible, and when it degrades
  - Heaps — the structure behind priority queues, min-heap vs. max-heap
  - Implementing a basic heap from scratch
- **Sorting & Searching**
  - Comparison-based sorts: bubble, insertion, selection (for intuition), merge sort, quicksort
  - Non-comparison sorts (brief mention: counting sort) — when they beat the O(n log n) lower bound and why
  - Binary search and its preconditions (sorted data)
  - Choosing the right sort/search for a given situation
- **Graphs**
  - Graph terminology and representations (adjacency list vs. adjacency matrix)
  - Breadth-first search (BFS) and depth-first search (DFS)
  - Shortest path basics: Dijkstra's algorithm (conceptual + implementation)
  - Where graphs show up in AI/ML contexts (computational graphs, knowledge graphs, dependency graphs)
- **Dynamic Programming**
  - Overlapping subproblems and optimal substructure — what makes a problem "DP-shaped"
  - Memoization (top-down) vs. tabulation (bottom-up)
  - Classic introductory DP problems (Fibonacci with memoization, coin change, simple knapsack)

## Weekly Study Plan

### Week 1 — Complexity Analysis

**Topics Covered:**
- Big O notation — what it measures and what it deliberately ignores (constants, lower-order terms)
- The difference between Big O (upper bound), Big Θ (tight bound), and Big Ω (lower bound) — Big O in practical everyday use
- Analyzing simple loops, nested loops, and basic recursive calls for time complexity
- Space complexity — measuring memory, not just time
- The common complexity classes and recognizing them in real code

**Tools / Technologies Used:**
- Python 3.x, `time` module for empirical timing comparisons

**Key Techniques / Concepts to Internalize:**
- Big O describes *growth rate*, not actual speed — an O(n) algorithm can be slower than an O(n²) one for small n; the notation only tells you what happens as n gets large. This distinction prevents a lot of premature optimization mistakes later.
- Nested loops over the same input generally multiply complexities (two nested O(n) loops → O(n²)), while sequential (non-nested) loops add (two separate O(n) loops → O(n), since constants drop out) — this single rule resolves most complexity-analysis confusion at this level.
- Complexity analysis is a tool for *comparing approaches before writing code*, not just an academic exercise — the habit of asking "what's the complexity here" before implementing something is what this week is actually building.

**Practical Tips:**
- Common mistake: assuming any loop inside a loop is automatically O(n²) — it depends on what each loop is actually iterating over; a loop inside a loop that runs a constant number of times is still effectively O(n).
- You understood this week if you can look at an unfamiliar 10-line function and state its time complexity within a minute, explaining your reasoning out loud.
- Empirically time a few of your own functions on inputs of increasing size (100, 1,000, 10,000 elements) and plot the results — seeing an O(n²) curve bend upward on an actual graph makes the abstract notation concrete.

**Recommended Resources:**

### Week 2 — Linear Data Structures

**Topics Covered:**
- Arrays vs. linked lists: how each is actually laid out in memory, and the resulting tradeoffs (random access vs. sequential access, insertion/deletion cost)
- Implementing a singly linked list from scratch (node class, insert, delete, traverse)
- Implementing a doubly linked list from scratch, and when the extra `prev` pointer is worth the overhead
- Stacks: LIFO behavior, implementing with a Python list, real use cases (undo functionality, expression evaluation, the call stack itself)
- Queues: FIFO behavior, implementing efficiently (why a plain Python list is a poor queue, and `collections.deque` is the right tool)

**Tools / Technologies Used:**
- Python 3.x, `collections.deque`

**Key Techniques / Concepts to Internalize:**
- Python's list is actually a dynamic array under the hood — this is *why* appending to the end is (amortized) O(1) but inserting/removing from the front is O(n), a fact you've likely used without knowing the reason until now.
- A linked list trades away random access (no `list[5]` in O(1)) for O(1) insertion/deletion at a known position — recognizing which operations your problem actually needs is what determines the right structure, not a general "linked lists are better/worse" instinct.
- The call stack you've been implicitly using with recursion since CS101 *is* a stack data structure — this week should make that connection explicit and concrete, not just an analogy.

**Practical Tips:**
- Common mistake: using `list.pop(0)` or `list.insert(0, x)` in a loop, which is O(n) per call and silently makes an otherwise-fine algorithm O(n²) — this is one of the most common real performance bugs in beginner-to-intermediate Python code.
- You understood this week if you can explain, without looking anything up, why `collections.deque` is the correct choice for a queue instead of a plain list.
- Implement the linked list operations fully by hand this week even though Python rarely requires you to — the goal is understanding what a dict/list is doing for you under the hood, not needing to reimplement it in daily work.

### Week 3 — Recursion & Backtracking

**Topics Covered:**
- Revisiting recursion from CS101 with complexity analysis: recursion trees, and an intuitive (not fully formal) sense of recurrence relations
- Backtracking as a pattern: make a choice, recurse, and undo ("backtrack") the choice if it leads nowhere
- Classic backtracking problems: generating permutations, generating combinations, solving a simple constraint-satisfaction puzzle (e.g., N-Queens or Sudoku-lite)
- Recognizing when a brute-force recursive approach is acceptable vs. when it needs pruning or a different technique entirely (foreshadowing dynamic programming, Week 8)

**Tools / Technologies Used:**
- Python 3.x

**Key Techniques / Concepts to Internalize:**
- A recursion tree is a genuinely useful way to *see* a recursive algorithm's complexity — drawing out how many calls happen at each depth is often more illuminating than trying to solve a recurrence relation formally at this stage.
- Backtracking is "try it, and if it fails, undo and try something else" — this three-part pattern (choose, explore, un-choose) is the actual mental template, and most backtracking bugs come from forgetting the "undo" step.
- Recognizing exponential blowup (many backtracking problems are technically O(2ⁿ) or worse) is important not to avoid recursion, but to know when a problem needs pruning, memoization, or a fundamentally different algorithm — the awareness itself is the goal this week, not yet the fix.

**Practical Tips:**
- Common mistake: mutating a shared data structure (like a list being built up during backtracking) without properly "undoing" that mutation before trying the next branch, leading to incorrect results that are hard to debug.
- You understood this week if you can implement a permutation generator from scratch using backtracking, correctly handling the undo step, without referencing an existing implementation.
- Draw the recursion tree by hand for at least one small backtracking problem (e.g., permutations of 3 items) before writing any code — seeing the branching structure on paper first makes the code much easier to reason about.

**Recommended Resources:**
- 📺 YouTube: Abdul Bari — "Backtracking Algorithm" (or similar clear conceptual explainer)
- 📘 Book: *Grokking Algorithms* — Chapter 4 (Quicksort, which reinforces recursion) plus any recursion-focused chapter available
- 📄 Docs: N/A — this topic is best learned via worked problems rather than a single reference doc; use the AI prompt and a problem set instead

Have me describe my choose/explore/un-choose logic in plain English before I write any code, then review my described approach for correctness."

### Week 4 — Trees & Binary Search Trees

**Topics Covered:**
- General tree terminology: root, leaf, parent/child, depth, height
- Binary trees vs. binary search trees (BSTs) — the ordering property that makes a BST useful
- BST operations: insertion, search, deletion (including the tricky case of deleting a node with two children)
- Tree traversal methods: in-order, pre-order, post-order (all via recursion), and level-order (via a queue — connecting back to Week 2)
- Balanced vs. unbalanced trees — a conceptual look at why an unbalanced BST degrades toward O(n) operations, without implementing a full self-balancing tree

**Tools / Technologies Used:**
- Python 3.x

**Key Techniques / Concepts to Internalize:**
- A BST's O(log n) average-case search only holds if the tree is reasonably balanced — a BST built by inserting already-sorted data degenerates into what's effectively a linked list (O(n) operations), which is a genuinely common real gotcha worth internalizing now rather than discovering via a slow production system later.
- In-order traversal of a BST visits nodes in sorted order — this isn't a coincidence, it's a direct consequence of the BST ordering property, and recognizing that connection deepens understanding of both traversal and the BST invariant itself.
- Level-order traversal is BFS on a tree (a specific case of the general graph BFS coming in Week 7) — using a queue to process nodes level by level is the same underlying pattern, just applied to a simpler structure first.

**Practical Tips:**
- Common mistake: implementing BST deletion without handling the two-children case correctly (the standard approach: replace with the in-order successor or predecessor, then delete that instead) — this is the single trickiest BST operation, and rushing it produces silently broken trees.
- You understood this week if you can implement in-order traversal recursively and explain, without checking, why it produces sorted output for a BST specifically (not for a general binary tree).
- Deliberately construct one balanced-looking BST and one badly unbalanced BST (by inserting already-sorted data) from the same values, and time search operations on both to feel the O(log n) vs O(n) difference directly.

### Week 5 — Hash Tables & Heaps

**Topics Covered:**
- How a hash table actually works: hash functions, buckets, and why a good hash function matters
- Collision handling strategies: chaining vs. open addressing (conceptual understanding of both)
- Why average-case O(1) lookup is possible, and the conditions under which it degrades toward O(n)
- Heaps: the structure behind priority queues, min-heap vs. max-heap, the array-based representation of a binary heap
- Core heap operations: insert (sift up), extract-min/max (sift down)

**Tools / Technologies Used:**
- Python 3.x, `heapq` module (for comparison after implementing your own)

**Key Techniques / Concepts to Internalize:**
- Python's `dict` is a hash table — this week should make what's actually happening under `dict[key] = value` and `dict[key]` concrete: hashing the key, finding the right bucket, handling any collision, rather than "just knowing dicts are fast."
- A heap is *not* a fully sorted structure — it only guarantees the min (or max) is at the root, with a weaker ordering everywhere else, which is exactly what makes insert and extract-min/max efficient (O(log n)) compared to keeping a fully sorted structure.
- Heaps and priority queues are effectively the same concept, and this is directly relevant to Dijkstra's algorithm next in Week 7 — always processing "the next closest/cheapest thing first" is a heap's whole purpose.

**Practical Tips:**
- Common mistake: assuming hash table lookups are *always* O(1) regardless of the hash function or load factor — with a poor hash function or too many collisions, worst-case degrades to O(n); "average case" is doing real work in that phrase.
- You understood this week if you can implement `heappush`/`heappop`-equivalent operations on a plain array-based binary heap from scratch and explain why sifting up/down maintains the heap property.
- Implement a basic hash table with chaining (a list of buckets, each holding a small list of key-value pairs) fully by hand this week — using `heapq` and `dict` freely afterward is fine, but only after you've built the mechanism once yourself.

### Week 6 — Sorting & Searching

**Topics Covered:**
- Comparison-based sorting for intuition: bubble sort, insertion sort, selection sort (implement at least one, understand why they're O(n²))
- Merge sort — divide and conquer, and why it guarantees O(n log n)
- Quicksort — partitioning, average-case vs. worst-case complexity, and why pivot choice matters
- A brief look at non-comparison sorts (counting sort) and why they can beat the O(n log n) comparison-sort lower bound under specific conditions
- Binary search — the precondition (sorted data), implementation, and common off-by-one pitfalls

**Tools / Technologies Used:**
- Python 3.x

**Key Techniques / Concepts to Internalize:**
- O(n log n) is a proven lower bound for *comparison-based* sorting — no comparison sort can do better in the general case, and understanding *why* (via the decision-tree argument, at least informally) explains why merge sort/quicksort's complexity isn't arbitrary, and why counting sort's better complexity only works because it isn't comparison-based.
- Quicksort's worst case is O(n²) despite average-case O(n log n) — this isn't a contradiction, it's entirely dependent on pivot choice and input structure (e.g., already-sorted data with a naive "always pick the first element" pivot strategy), which is a genuinely important practical caveat.
- Binary search's precondition — the data must already be sorted — is the single most commonly overlooked detail; applying binary search to unsorted data doesn't error, it just silently returns wrong answers.

**Practical Tips:**
- Common mistake: getting binary search's boundary conditions wrong (`low <= high` vs `low < high`, and how `mid` is updated) — walk through a tiny 3-4 element example by hand every time you're unsure, rather than guessing.
- You understood this week if you can implement merge sort and quicksort from scratch without referencing an existing implementation, and explain in one sentence why each is O(n log n) on average.
- Empirically time your from-scratch sorts against Python's built-in `sorted()` (which uses Timsort) on increasingly large inputs — Python's built-in will win by a wide margin, and understanding *why* (a highly optimized, hybrid, adaptive algorithm) is a useful, humbling data point.

### Week 7 — Graphs

**Topics Covered:**
- Graph terminology: directed vs. undirected, weighted vs. unweighted, cyclic vs. acyclic
- Graph representations: adjacency list vs. adjacency matrix, and the space/time tradeoffs between them
- Breadth-first search (BFS) — implementation using a queue, use cases (shortest path in unweighted graphs, level-order traversal generalized)
- Depth-first search (DFS) — implementation using recursion or an explicit stack, use cases (cycle detection, connectivity, topological sorting)
- Dijkstra's algorithm — finding shortest paths in a weighted graph, using a heap (directly building on Week 5)
- Where graphs show up in AI/ML: computational graphs (MA102's backprop), knowledge graphs, dependency graphs in build systems

**Tools / Technologies Used:**
- Python 3.x, `heapq` (for Dijkstra)

**Key Techniques / Concepts to Internalize:**
- BFS and DFS explore a graph in fundamentally different orders (level-by-level vs. as-deep-as-possible-first), and that difference in *order* is exactly what makes each suited to different problems (BFS for shortest unweighted paths, DFS for exhaustive exploration and structural questions like cycle detection).
- Dijkstra's algorithm is, at its core, "always process the currently-cheapest-known unvisited node next" — which is precisely why a min-heap (Week 5) is the natural underlying structure, not an arbitrary implementation choice.
- Recognizing when a problem is secretly a graph problem is a genuine skill — dependency resolution, "shortest sequence of steps," and network-like relationships between entities are all graph problems even when they aren't presented with explicit "nodes and edges" language.

**Practical Tips:**
- Common mistake: forgetting to track visited nodes in BFS/DFS on a graph with cycles, causing infinite loops — always maintain an explicit visited set, unlike tree traversal where this usually isn't needed.
- You understood this week if you can implement BFS and DFS from scratch on an adjacency-list representation, and explain — without looking anything up — which one you'd use to find the shortest path in an unweighted graph, and why.
- Implement Dijkstra's algorithm by hand once, using your own heap from Week 5 (or `heapq`), rather than only ever calling a library function — the goal is understanding the "always take the cheapest next step" logic concretely.

### Week 8 — Dynamic Programming & Wrap-Up

**Topics Covered:**
- Recognizing when a problem is "DP-shaped": overlapping subproblems and optimal substructure
- Memoization (top-down): adding a cache to a recursive solution to avoid recomputing overlapping subproblems
- Tabulation (bottom-up): building up a solution iteratively from base cases
- Classic introductory DP problems: Fibonacci with memoization (revisited from MA102), coin change, a simple 0/1 knapsack
- Course wrap-up: reviewing which structure/algorithm to reach for given a problem's shape, and integrating everything into the Main Project

**Tools / Technologies Used:**
- Python 3.x

**Key Techniques / Concepts to Internalize:**
- Memoization is literally "recursion plus a cache" — if you can already write the plain recursive solution to a problem, adding memoization is a small, mechanical change, not a fundamentally different technique. This reframing removes a lot of the intimidation around DP.
- Tabulation and memoization solve the exact same subproblems, just in opposite order (bottom-up building vs. top-down with caching) — recognizing they're two directions on the same idea, not unrelated techniques, is the actual "aha" of this week.
- The jump from exponential (plain recursion, Week 3's backtracking-adjacent blowup) to polynomial (with memoization) time by caching overlapping subproblems is one of the most dramatic, satisfying complexity improvements you'll see in this course — and it directly closes the loop on the exponential-blowup concern raised in Week 3.

**Practical Tips:**
- Common mistake: trying to apply memoization to a problem that doesn't actually have overlapping subproblems (in which case there's nothing to cache, and it won't help) — always check first whether the same subproblem is genuinely being recomputed.
- You understood this week if you can take the plain recursive Fibonacci function, identify exactly which subproblems are being recomputed, and add memoization correctly without referencing an example.
- This week doubles as course integration — the goal isn't mastering all of DP (that's a much bigger topic with a long tail of harder problems) but recognizing the pattern and being unafraid of it going forward.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "Dynamic Programming - Learn to Solve Algorithmic Problems"
- 📘 Book: *Grokking Algorithms* — Chapter 9 (Dynamic Programming)
- 📄 Docs: N/A — best learned through worked problems; use the AI prompt and problem practice instead
- 🤖 AI Prompt: "Give me the plain recursive Fibonacci function. Have me identify exactly which calls are being recomputed unnecessarily for fib(6), then have me add memoization myself and check whether my implementation actually eliminates the redundant calls."

## Main Project

**Project: Grid Pathfinding Visualizer**

A tool that visualizes and compares BFS, DFS, and Dijkstra's algorithm finding a path across a 2D grid with obstacles and (optionally) varying terrain costs.

**Requirements:**
- Represent a 2D grid as a graph (each open cell connects to its valid neighbors), supporting configurable obstacles (impassable cells).
- Implement BFS, DFS, and Dijkstra's algorithm (using a heap) to find a path from a start cell to an end cell.
- For Dijkstra specifically, support at least two different terrain costs (e.g., "normal" cells cost 1, "difficult terrain" cells cost 3) so the weighted-graph behavior is actually exercised, not just a relabeled BFS.
- Visualize each algorithm's result: the final path found, and ideally the order in which cells were explored (even a simple Matplotlib grid coloring by visit order is enough — this doesn't need to be a live animation).
- Report path length/cost and number of cells explored for each algorithm on the same grid, and write a short comparison in the project's README.

**Stretch Goals:**
- Implement A* search (Dijkstra + a heuristic) and compare its number of explored cells against plain Dijkstra on the same grid.
- Generate random grids with random obstacles and terrain costs, and run all algorithms across several grids to build a more general comparison rather than a single example.
- Add a simple text-based or Matplotlib-based step-by-step animation of the search frontier expanding.

**What "Done" Looks Like:**
- All three algorithms run correctly on at least one non-trivial grid with obstacles, producing valid paths (or correctly reporting "no path exists" when the end is unreachable).
- You have at least one clear visualization comparing the algorithms' explored-cell counts or paths on the same grid.
- You can explain, without notes, why BFS finds the shortest path on an unweighted grid but can produce a suboptimal path once terrain costs are introduced, while Dijkstra handles both correctly.

## Self-Check Before Moving On

Before starting CS104, you should honestly be able to say:

- I can state the time complexity of an unfamiliar function within a minute or two, explaining my reasoning.
- I've implemented a linked list, a BST, a hash table, and a heap from scratch, not just used Python's built-in equivalents.
- I can explain the difference between BFS and DFS, and correctly choose between them for a given problem.
- I can recognize when a problem is "DP-shaped" (overlapping subproblems, optimal substructure) and add memoization to a plain recursive solution.
- I completed the Grid Pathfinding Visualizer and can explain why each algorithm behaves differently on weighted vs. unweighted grids.

If any of these feel shaky, that's a signal to spend a few more days here — CS104's software design work assumes you're comfortable choosing and implementing the right structure for a problem, not still learning what the structures are.
