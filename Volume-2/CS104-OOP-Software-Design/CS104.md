# CS104 — OOP & Software Design

## Why This Matters

CS101 gave you just enough OOP to read and write simple classes. This course is where that becomes real software design: structuring code so it can grow, be tested, and be changed without everything breaking. Real ML/AI codebases — training pipelines, data processing frameworks, agent systems — are software projects with the same design pressures as any other production codebase. Skipping real software design skills is how self-taught engineers end up with 500-line scripts that work once and can't be safely modified.

## Prerequisites

- **CS101 (Programming Foundations)** — comfortable with the light OOP introduction from Week 7 (classes, `__init__`, instance methods).
- **CS103 (Data Structures & Algorithms)** — this course reuses several of CS103's custom data structures as concrete examples for practicing design.

## Where This Is Used Later

- **ML102 (Feature Engineering & Data Pipelines)** — building a reusable, extensible data pipeline is a direct software design problem; this course's Main Project previews exactly that.
- **DL102 (Deep Learning with PyTorch)** — PyTorch's own API (subclassing `nn.Module`, composing layers) is built entirely on the inheritance and composition patterns from this course.
- **AIE101–AIE105 (AI Engineering & Production Systems)** — every one of these courses involves building systems (not scripts) meant to be maintained, tested, and extended; this course is the foundation for treating that work like real software.
- **Any collaborative or long-lived project from here on** — good software design is what makes code reviewable and safely changeable by you-in-six-months, not just by other people.

## Full Topic Breakdown

- **OOP Pillars, Properly**
  - Encapsulation — bundling data and behavior, and controlling access (naming conventions for "private," properties)
  - Abstraction — hiding implementation details behind a clear interface
  - Inheritance — sharing behavior across related classes, `super()`, method resolution order (conceptual)
  - Polymorphism — treating different types uniformly through a shared interface
  - Dunder (magic) methods beyond `__init__`/`__repr__`: `__eq__`, `__len__`, `__iter__`, `__enter__`/`__exit__`
  - Properties (`@property`) — computed attributes and controlled access without breaking encapsulation
- **Inheritance & Polymorphism in Practice**
  - When inheritance is the right tool, and its common misuses
  - Abstract base classes (`abc` module) — defining an interface that subclasses must implement
  - Method overriding, and calling the parent implementation via `super()`
  - Duck typing — Python's structural approach to polymorphism vs. strict interface inheritance
- **Composition & Design Principles**
  - Composition over inheritance — building behavior by combining objects rather than deep class hierarchies
  - SOLID principles, introduced practically (not as abstract theory): Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion
  - Recognizing "code smells" that suggest a design needs to change (God objects, deep inheritance chains, tight coupling)
- **Common Design Patterns**
  - Strategy pattern — swapping algorithms/behaviors at runtime
  - Factory pattern — centralizing object creation logic
  - Observer pattern — decoupling event producers from consumers
  - Singleton pattern — and why it's often overused/misused, with a note on when it's actually appropriate
- **Modern Python Practices**
  - Type hints: function signatures, `Optional`, `Union`, generics (basic level)
  - `dataclasses` — reducing boilerplate for data-holding classes
  - Static type checking with `mypy` (introductory use)
- **Testing & Testable Design**
  - Why untested code is a design liability, not just a quality issue
  - Writing unit tests with `pytest`
  - Test structure: arrange/act/assert, fixtures (basic use)
  - Designing code to be testable in the first place (dependency injection at a basic level, avoiding hidden global state)
- **Packaging & Project Structure**
  - Organizing a multi-file/multi-module Python project properly
  - `__init__.py`, package structure, relative vs. absolute imports
  - A basic `pyproject.toml`/`setup.py` for making a project installable
  - Writing a project README that actually helps someone (including future-you) use the code

### Week 1 — OOP Pillars, Properly

**Topics Covered:**
- Encapsulation: naming conventions (`_protected`, `__private`), why Python's "privacy" is convention-based, not enforced
- Abstraction: designing a clear public interface that hides messy implementation details
- Dunder methods beyond `__init__`: `__repr__` vs `__str__`, `__eq__`, `__len__`, `__iter__`
- `@property` — exposing computed or validated attributes without forcing callers to call a method explicitly

**Tools / Technologies Used:**
- Python 3.x, VS Code

**Key Techniques / Concepts to Internalize:**
- Python's "privacy" (`_single_underscore`, `__double_underscore`) is a convention and a mild technical speed bump (name mangling for double-underscore), not real access control — understanding this prevents both over-relying on it for security and under-using it as a genuinely useful signal to other developers (and future you).
- Implementing `__eq__` or `__iter__` on your own class is how you make custom objects behave like "real" Python objects — usable in `==` comparisons, `for` loops, `sorted()`, etc. — instead of needing special-case handling everywhere they're used.
- `@property` lets you start with a plain attribute and later add validation or computed logic without changing the class's public interface — recognizing *when* this matters (you don't need it for every attribute) is as important as knowing the syntax.

**Practical Tips:**
- Common mistake: implementing `__eq__` without also implementing `__hash__` when the object needs to be used in a set or as a dict key — Python disables default hashing once you define `__eq__`, which causes confusing errors later if forgotten.
- You understood this week if you can add `__repr__`, `__eq__`, and `__iter__` to a custom class and have it behave naturally in `print()`, `==` comparisons, and a `for` loop.
- Don't over-encapsulate every attribute behind a property "just in case" — start with plain attributes and add a property only when you actually need validation or computed behavior; premature encapsulation is its own kind of complexity.

### Week 2 — Inheritance & Polymorphism in Practice

**Topics Covered:**
- When inheritance genuinely models an "is-a" relationship vs. when it's being misused for convenience
- Abstract base classes with the `abc` module — defining a required interface that subclasses must implement
- Overriding methods, and correctly calling the parent implementation via `super()`
- Duck typing — Python's "if it walks like a duck and quacks like a duck" approach to polymorphism, and how it differs from strict interface-based polymorphism in other languages

**Tools / Technologies Used:**
- Python 3.x, `abc` module

**Key Techniques / Concepts to Internalize:**
- Inheritance should model a genuine "is-a" relationship (a `Dog` is an `Animal`) — reaching for inheritance purely to reuse some code without that relationship actually holding is one of the most common real design mistakes, and is exactly what "composition over inheritance" (Week 3) pushes back against.
- An abstract base class isn't about restricting what you can do — it's a contract that says "any subclass of this must implement these specific methods," which becomes valuable the moment more than one type needs to be used interchangeably by other code.
- Duck typing means Python often doesn't *require* formal inheritance for polymorphism to work — if two unrelated classes both implement a `.speak()` method, code that calls `.speak()` on either works fine without a shared base class. Knowing when to use this flexibility vs. when a formal interface (abstract base class) adds real value is a genuine design judgment call.

**Practical Tips:**
- Common mistake: building a deep inheritance chain (`Animal` → `Mammal` → `Dog` → `Labrador` → `ServiceDog`) speculatively, "in case it's needed later" — shallow hierarchies with composition for the rest are almost always easier to maintain than deep ones.
- You understood this week if you can look at two unrelated classes with a similarly-named method and explain whether that's duck typing working correctly, or a coincidence that should be formalized with a shared abstract base class.
- Implement at least one abstract base class this week with a genuinely required method (not just an example) — the point isn't syntax, it's feeling what it's like when a subclass "forgets" to implement something and Python enforces it.

**Recommended Resources:**
- 📺 YouTube: Corey Schafer — "Python OOP Tutorial 3: Inheritance"
- 📘 Book: *Fluent Python* — Chapter 13 (Interfaces, Protocols, and ABCs)
- 📄 Docs: [docs.python.org/3/library/abc.html](https://docs.python.org/3/library/abc.html)
- 🤖 AI Prompt: "Give me a scenario with 3 different classes (like Circle, Square, Triangle) that all need an area() method. Have me design this two ways — with an abstract base class, and with pure duck typing (no shared base class) — and explain the tradeoffs of each approach for this specific case."

**This Week's Mini Exercise:**
Design an abstract base class `Shape` with an abstract method `area()`, implement at least 3 concrete subclasses (e.g. `Circle`, `Rectangle`, `Triangle`), and write a function that takes a list of mixed `Shape` objects and returns the total area — confirming polymorphism means that function never needs to know or care which specific shape it's looking at.

### Week 3 — Composition & Design Principles

**Topics Covered:**
- Composition over inheritance: building objects by combining smaller, focused objects instead of deep class hierarchies
- SOLID principles, introduced practically through small refactoring examples rather than abstract definitions:
  - Single Responsibility — a class should have one reason to change
  - Open/Closed — open for extension, closed for modification
  - Liskov Substitution — a subclass should be usable anywhere its parent is expected
  - Interface Segregation — many small, specific interfaces beat one large general one
  - Dependency Inversion — depend on abstractions, not concrete implementations
- Recognizing common "code smells": God objects (classes that do too much), tight coupling, deep inheritance chains

**Tools / Technologies Used:**
- Python 3.x

**Key Techniques / Concepts to Internalize:**
- "Composition over inheritance" doesn't mean inheritance is bad — it means inheritance should be the exception, reserved for genuine is-a relationships, while most code reuse should happen by giving an object other objects to delegate to (has-a relationships). This is the single biggest shift in design instinct this course aims to build.
- SOLID principles are easiest to internalize as answers to "why did this refactor make the code better," not as rules to apply in the abstract — each principle should be introduced alongside a concrete before/after example showing a real problem it solves.
- A "God object" (one class doing configuration, data storage, business logic, and I/O all at once) is the most common real symptom of missing Single Responsibility — if you can't summarize a class's job in one sentence without "and," that's a signal.

**Practical Tips:**
- Common mistake: applying every SOLID principle rigidly to every piece of code regardless of actual complexity — these are guidelines for managing complexity as it grows, not rules that improve a genuinely simple 20-line script.
- You understood this week if, given a class that's clearly doing too much, you can identify which specific responsibilities should be split out and roughly how.
- Practice this week by refactoring, not just reading — take a deliberately badly-designed class (write one yourself, or use the AI prompt below) and split it apart using composition and SOLID principles, rather than only studying definitions.

**Recommended Resources:**
- 📺 YouTube: ArjanCodes — "SOLID Principles in Python" (or similar practical, refactoring-focused Python SOLID content)
- 📘 Book: *Fluent Python* — Chapter 24 (Class Metaprogramming, skim) — but more directly, search for "Python composition over inheritance" articles for practical grounding
- 📄 Docs: N/A — this topic is best learned via refactoring exercises rather than a single reference; use the AI prompt and hands-on practice instead
- 🤖 AI Prompt: "Give me a deliberately badly-designed 'God object' class — something like a `UserManager` that handles validation, database access, email sending, and logging all in one class. Have me identify which responsibilities should be split out, then review my proposed refactor."

**This Week's Mini Exercise:**
Take (or write) a God object class doing at least 3 unrelated jobs (e.g. a `ReportGenerator` that fetches data, formats it, and writes it to a file), refactor it into 3+ smaller classes each with a single responsibility, composed together, and write a short before/after comparison explaining what specifically improved.

### Week 4 — Common Design Patterns

**Topics Covered:**
- Strategy pattern — encapsulating interchangeable algorithms/behaviors behind a common interface, selectable at runtime
- Factory pattern — centralizing and abstracting away object creation logic
- Observer pattern — decoupling something that produces events from the things that need to react to them
- Singleton pattern — what it is, why it's frequently overused or misused, and the narrow cases where it's genuinely appropriate

**Tools / Technologies Used:**
- Python 3.x

**Key Techniques / Concepts to Internalize:**
- Design patterns are named solutions to *recurring* problems, not a checklist to apply everywhere — the value this week is recognizing "oh, this is a Strategy pattern situation" when it naturally arises, not forcing a pattern onto code that doesn't need it.
- Strategy pattern is directly useful and common in ML/data contexts — swapping out different preprocessing steps, different model evaluation metrics, or different data loading strategies behind a common interface is exactly this pattern, and previews the Main Project's design.
- Singleton is worth learning primarily as a caution — it introduces global mutable state under a respectable-sounding name, and in Python specifically, module-level state or dependency injection often solves the same problem more cleanly without the downsides.

**Practical Tips:**
- Common mistake: reaching for a Factory pattern when a plain function or even direct instantiation would be simpler — factories earn their complexity when object creation is genuinely complex or needs to be decided dynamically, not by default.
- You understood this week if, given a scenario needing interchangeable behavior at runtime (e.g. different sorting strategies, different notification methods), you can identify Strategy pattern as the fit and implement it without over-engineering.
- Implement each pattern once, from a real small scenario, rather than only reading about it in the abstract — patterns learned only conceptually rarely transfer to recognizing them in real code later.

**Recommended Resources:**
- 📺 YouTube: ArjanCodes — "Design Patterns in Python" series (Strategy, Factory, Observer specifically)
- 📘 Book: *Head First Design Patterns* (language-agnostic, but concepts map cleanly to Python) — Strategy, Factory, Observer chapters
- 📄 Docs: [refactoring.guru — Design Patterns](https://refactoring.guru/design-patterns/python) (Python-specific examples)
- 🤖 AI Prompt: "Give me a real scenario where I need to support 3 different sorting strategies for a list of items, selectable at runtime by the user. Have me implement this using the Strategy pattern, then review my implementation and point out anything I over-engineered or under-designed."

**This Week's Mini Exercise:**
Implement a small notification system using the Strategy pattern (interchangeable "send via email," "send via SMS," "send via console log" strategies behind a common interface) and the Observer pattern (a simple event system where multiple observers react to a single "event occurred" trigger) — two small, separate implementations, each demonstrating one pattern clearly.

### Week 5 — Type Hints, Dataclasses & Modern Python Practices

**Topics Covered:**
- Type hints: annotating function parameters and return types, `Optional`, `Union` (or the newer `|` syntax), basic generics (`list[int]`, `dict[str, int]`)
- `dataclasses` — the `@dataclass` decorator, reducing boilerplate for classes that primarily hold data
- Static type checking with `mypy` — running it, reading its output, and fixing type errors it catches
- When type hints and dataclasses genuinely help vs. when they're unnecessary ceremony for a truly small script

**Tools / Technologies Used:**
- Python 3.x, `mypy`, `dataclasses` (standard library)

**Key Techniques / Concepts to Internalize:**
- Type hints don't change Python's runtime behavior at all by default — they're documentation that a tool (`mypy`) can check for you, catching a category of bugs (passing the wrong type) before you ever run the code. Understanding this "hints are for tools and humans, not the interpreter" framing prevents confusion about what they actually do.
- `@dataclass` automatically generates `__init__`, `__repr__`, and `__eq__` for you based on declared fields — this is directly related to Week 1's dunder methods, just auto-generated instead of hand-written, and recognizing that connection makes dataclasses feel like a shortcut rather than new magic.
- Type hints pay off disproportionately once a codebase has multiple files/functions calling each other — a small standalone script may not need them, but anything you intend to maintain, extend, or that others (including future you) will read benefits significantly.

**Practical Tips:**
- Common mistake: adding type hints but never actually running `mypy` — the hints alone don't catch anything; the checker is what does the actual work, so it needs to be part of your workflow, not just decoration.
- You understood this week if you can convert a plain class with a hand-written `__init__`/`__repr__`/`__eq__` into an equivalent `@dataclass` and confirm the behavior is identical.
- Don't over-annotate a small personal script "for completeness" — practice type hints deliberately on projects worth it (like this course's Main Project or the CS103 project), while it's fine to skip them for a quick throwaway script.

**Recommended Resources:**
- 📺 YouTube: ArjanCodes — "Python Type Hints" and "Dataclasses in Python"
- 📘 Book: *Fluent Python* — Chapter 15 (More About Type Hints)
- 📄 Docs: [docs.python.org/3/library/dataclasses.html](https://docs.python.org/3/library/dataclasses.html), [mypy.readthedocs.io](https://mypy.readthedocs.io/)
- 🤖 AI Prompt: "I have a plain Python class with a hand-written __init__, __repr__, and __eq__ representing a 'Point' with x and y coordinates. Walk me through converting it to a @dataclass, and explain exactly what functionality I get for free versus what I had to write by hand before."

**This Week's Mini Exercise:**
Take 3 classes from earlier in this course (or CS103) that are primarily data-holding, convert them to `@dataclass`, add full type hints to their fields and any related functions, then run `mypy` on the resulting code and fix any type errors it reports.

### Week 6 — Testing & Testable Design

**Topics Covered:**
- Why untested code is a design liability — untested code resists refactoring because there's no safety net confirming behavior didn't break
- Writing unit tests with `pytest`: test discovery, `assert`-based tests, running a test suite
- Test structure: arrange/act/assert as a mental template for writing clear tests
- Basic use of `pytest` fixtures for shared setup
- Designing code to be testable: avoiding hidden global state, injecting dependencies (e.g. passing a data source in rather than hardcoding a file path inside a function)

**Tools / Technologies Used:**
- Python 3.x, `pytest`

**Key Techniques / Concepts to Internalize:**
- A test suite is a specification of behavior you can re-check automatically, at any time, in seconds — the value isn't "catching bugs once," it's making every future change safe to attempt, which is precisely what enables the kind of refactoring practiced in Week 3.
- Testable design and good OOP design are the same skill wearing different names — a class with hidden global state, tightly coupled dependencies, or too many responsibilities (Week 3's God object) is *exactly* the kind of code that's hard to test, which is not a coincidence.
- Arrange/act/assert is a simple enough template to feel almost too basic, but consistently structuring tests this way (set up the situation, perform the action, check the result) makes tests dramatically easier to read and debug when they fail months later.

**Practical Tips:**
- Common mistake: writing a test that both sets up complex state and checks many unrelated things at once — if a test's name can't describe one specific behavior being verified, it's probably testing too much at once.
- You understood this week if you can take an untested function, write a handful of `pytest` tests covering its normal case and at least one edge case, and explain what each test is actually verifying.
- Practice designing for testability directly: take a function that reads a hardcoded file path internally, and refactor it to accept a file path (or even better, already-loaded data) as a parameter instead — notice how much easier it becomes to test without touching the real filesystem.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "Testing Your Python Code with Pytest"
- 📘 Book: *Python Testing with pytest* by Brian Okken — Chapters 1–3
- 📄 Docs: [docs.pytest.org/en/stable/getting-started.html](https://docs.pytest.org/en/stable/getting-started.html)
- 🤖 AI Prompt: "Here's a small untested function that calculates shipping cost based on weight and destination [describe or provide one]. Have me write pytest tests covering the normal case, a boundary case, and an invalid input case, then review my tests for whether each one is checking exactly one clear behavior."

## Main Project

**Project: Modular ETL Pipeline Framework**

A small, extensible framework for building data pipelines (Extract → Transform → Load steps), designed specifically to showcase composition, the Strategy pattern, type hints, and real test coverage — and to directly preview the kind of data pipeline work coming in ML102.

**Requirements:**
- Define abstract base classes (or protocols) for `Extractor`, `Transformer`, and `Loader` — each with a clear, minimal required interface.
- Implement at least 2 concrete extractors (e.g. from CSV, from JSON), 2 transformers (e.g. filtering rows, normalizing a column), and 2 loaders (e.g. write to CSV, print summary to console).
- Build a `Pipeline` class that composes an extractor, a list of transformers, and a loader (Strategy pattern in action — swap any piece without touching the `Pipeline` class itself).
- Fully type-hint the framework and confirm it passes `mypy` with no errors.
- Write a `pytest` test suite covering each extractor, transformer, and loader independently, plus at least one full end-to-end pipeline test.
- Package the project properly: `pyproject.toml`, installable locally, with a clear README.

**Stretch Goals:**
- Add a simple plugin-discovery mechanism (e.g. a registry dict mapping names to transformer classes) so new transformers can be added and selected by name without editing the `Pipeline` class.
- Add an Observer-pattern-based logging/progress system that reports on each pipeline stage as it runs, without the `Pipeline` class needing to know about logging directly.
- Add a `--config` option (using `argparse`, from CS101) that builds a pipeline from a JSON/YAML config file specifying which extractor/transformers/loader to use.

**What "Done" Looks Like:**
- You can swap any extractor, transformer, or loader for a different implementation without modifying the `Pipeline` class itself — proving the Open/Closed principle is actually being honored, not just claimed.
- `mypy` and the full `pytest` suite both pass cleanly.
- The project installs locally via `pip install -e .` and can be imported/run from any directory.
- Pushed to this repo's `Projects/` folder with a README explaining the architecture and how to add a new extractor/transformer/loader.
