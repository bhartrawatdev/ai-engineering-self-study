# MA101 — Linear Algebra for ML

## Why This Matters

Neural networks are, mechanically, sequences of matrix multiplications and vector transformations — every weight matrix, every embedding, every gradient is a linear algebra object first and an "AI concept" second. Without this course, DL101 (Neural Networks From Scratch) becomes memorizing formulas instead of understanding what's actually happening when data flows through a network. This course exists to make matrices and vectors as intuitive as the arithmetic you already know, so the AI-specific courses can focus on what's new instead of re-deriving the math underneath.

## Prerequisites

- **CS101 (Programming Foundations)** — specifically comfortable with the Week 8 NumPy preview; this course uses NumPy throughout to make every concept concrete instead of purely symbolic.

## Where This Is Used Later

- **MA102 (Calculus & Optimization Basics)** — gradients are vectors, and the Jacobian/Hessian machinery there is built directly on this course's matrix foundations.
- **ML101 (Classical Machine Learning)** — linear regression, PCA, and most classical ML algorithms are linear algebra applied directly; this course does the heavy lifting so ML101 can focus on the ML framing.
- **DL101 (Neural Networks From Scratch)** — every layer of a neural network is a matrix multiplication followed by a nonlinearity; you cannot implement backprop by hand without this.
- **NLP102 (Transformer Architecture Deep Dive)** — attention mechanisms are matrix operations on vector representations; this course is the literal prerequisite vocabulary.

## Full Topic Breakdown

- **Vectors**
  - Vectors as arrows, as points, and as ordered lists of numbers — three views of the same object
  - Vector addition, scalar multiplication, geometric interpretation
  - Dot product — algebraic definition and geometric meaning (projection, angle)
  - Vector norms: L1, L2, L∞, and why "distance" isn't a single fixed concept
- **Matrices**
  - Matrices as collections of vectors, and as linear transformations
  - Matrix addition, scalar multiplication, matrix multiplication (and why it's defined the way it is)
  - Transpose, symmetric matrices
  - Identity matrix, special matrices (diagonal, triangular)
- **Linear Transformations & Systems of Equations**
  - A matrix as a function that transforms space (rotation, scaling, shearing, projection)
  - Systems of linear equations as `Ax = b`
  - Solving systems: Gaussian elimination (conceptual), when a unique solution exists
- **Determinants, Inverses & Rank**
  - Determinant — what it measures geometrically (area/volume scaling), and when it's zero
  - Matrix inverse — what it means for a transformation to be "undoable," and when it doesn't exist
  - Rank — the actual "information content" of a matrix, and rank-deficiency
- **Eigenvalues, Eigenvectors & Diagonalization**
  - What an eigenvector actually is — a direction a transformation doesn't rotate, only scales
  - Computing eigenvalues/eigenvectors (conceptually, and via NumPy in practice)
  - Diagonalization and why it matters (repeated transformations, understanding a matrix's "shape")
- **Norms, Orthogonality & Dimensionality Reduction**
  - Orthogonal and orthonormal vectors/bases
  - Projections onto subspaces
  - Singular Value Decomposition (SVD) — conceptual understanding and practical use
  - Principal Component Analysis (PCA) as an application of eigenvectors/SVD — a first bridge into ML101

## Weekly Study Plan

### Week 1 — Vectors & Vector Operations

**Topics Covered:**
- Vectors as arrows, points, and ordered lists — reconciling the three mental models
- Vector addition and scalar multiplication, geometrically and algebraically
- The dot product: algebraic formula and geometric meaning (projection, angle between vectors)
- Vector norms: L1, L2, L∞

**Tools / Technologies Used:**
- Python 3.x, NumPy, Matplotlib (for 2D visualization)

**Key Techniques / Concepts to Internalize:**
- A vector isn't "a list of numbers that happens to have geometric meaning" — the list-of-numbers view and the arrow-in-space view are the same object, and fluently switching between them is what makes later ML concepts (embeddings, gradients) feel concrete instead of abstract.
- The dot product being zero means the vectors are perpendicular — this single fact underlies orthogonality, projections, and (much later) why certain neural network initialization schemes work.
- L1 vs. L2 norm isn't just "two ways to measure length" — L1 (sum of absolute values) and L2 (Euclidean distance) behave differently under optimization, which is why you'll see both again as regularization terms in ML101.

**Practical Tips:**
- Common mistake: treating the dot product as "just an algebra formula" without connecting it to `‖a‖‖b‖cos(θ)` — both views are needed, not just the one that's easier to compute.
- You understood this week if you can look at two 2D vectors and estimate, without calculating, whether their dot product is positive, negative, or near zero, based on the angle between them.
- Plot everything this week. Linear algebra taught purely symbolically, without ever seeing a vector on a graph, builds much weaker intuition than the same material paired with visualization.

### Week 2 — Matrices & Matrix Operations

**Topics Covered:**
- Matrices as collections of vectors (rows or columns) and as linear transformations
- Matrix addition, scalar multiplication
- Matrix multiplication — the mechanics, and why the "row times column" definition is the one that makes composition of transformations work
- Transpose, symmetric matrices
- The identity matrix and other special matrices (diagonal, upper/lower triangular)

**Tools / Technologies Used:**
- Python 3.x, NumPy, Matplotlib

**Key Techniques / Concepts to Internalize:**
- Matrix multiplication is not element-wise (that's a different, less common operation) — it's defined the specific way it is *because* it represents composing two transformations, and that's the framing that makes the dimension-matching rule (`(m×n)(n×p)`) make sense instead of feeling arbitrary.
- Multiplying a matrix by a vector is applying a transformation to that vector — this is the single idea that will make neural network forward passes intuitive later (each layer is "apply this matrix's transformation, then a nonlinearity").
- Matrix multiplication is not commutative (`AB ≠ BA` in general) — internalizing *why*, geometrically (order of transformations matters), prevents a very common source of later bugs when implementing anything by hand.

**Practical Tips:**
- Common mistake: confusing element-wise multiplication (`*` in NumPy) with true matrix multiplication (`@` or `np.matmul`) — this exact confusion causes silent, hard-to-debug shape errors in real ML code.
- You understood this week if you can predict the output shape of a matrix multiplication (`(m,n) @ (n,p) → (m,p)`) without running code, for several examples.
- Spend real time visualizing a 2×2 matrix as a transformation of the plane (where do the basis vectors end up?) — this is the single most useful mental model 3Blue1Brown's series builds, and it pays off for the rest of the course.

**Recommended Resources:**
- 📺 YouTube: 3Blue1Brown — "Essence of Linear Algebra," Chapters 3–4 (Linear transformations, Matrix multiplication as composition)
- 📘 Book: *Mathematics for Machine Learning* — Chapter 2.2–2.3 (Matrices)
- 📄 Docs: [numpy.org — `numpy.matmul` vs `numpy.multiply`](https://numpy.org/doc/stable/reference/generated/numpy.matmul.html)
- 🤖 AI Prompt: "Give me two small matrices where the dimensions could plausibly be multiplied in either order but only one order is valid. Ask me to identify which multiplication is valid and predict the resulting shape before I compute it."

**This Week's Mini Exercise:**

### Week 3 — Linear Transformations & Systems of Equations

**Topics Covered:**
- A matrix as a function transforming space: rotation, scaling, shearing, projection matrices in 2D
- Systems of linear equations expressed as `Ax = b`
- Gaussian elimination — the conceptual process of row-reducing to solve a system
- When a system has a unique solution, no solution, or infinitely many

**Tools / Technologies Used:**
- Python 3.x, NumPy (`numpy.linalg.solve`), Matplotlib

**Key Techniques / Concepts to Internalize:**
- `Ax = b` is the same "matrix as transformation" idea from Week 2, just asked in reverse: "what input, transformed by A, produces this output?" — recognizing systems of equations as an inverse-transformation question, not a separate topic, ties the whole course together.
- Gaussian elimination isn't just a mechanical algorithm to memorize — it's systematically simplifying the system until the answer is obvious, and understanding *why* each row operation is valid (it doesn't change the solution set) matters more than speed at doing it by hand.
- Whether a system has a unique solution connects directly to concepts you'll meet properly next week (rank, determinant) — a system with no unique solution is really telling you something about the matrix itself, not just about that particular `b`.

**Practical Tips:**
- Common mistake: solving small systems by hand without connecting it back to the geometric picture (lines/planes intersecting) — always sanity-check a 2D system by picturing the two lines and where they'd cross.
- You understood this week if you can look at a 2×2 system and predict, before solving, whether it has a unique solution just by checking if the two equations describe parallel lines.
- Use `numpy.linalg.solve` to check your by-hand work this week, not to skip doing it by hand — the point is building the intuition, not just getting the right number.

**Recommended Resources:**
- 📺 YouTube: 3Blue1Brown — "Essence of Linear Algebra," Chapter 6-7 (Linear systems, Nonsquare matrices)
- 📘 Book: *Mathematics for Machine Learning* — Chapter 2.1 (Systems of Linear Equations)
- 📄 Docs: [numpy.org — `numpy.linalg.solve`](https://numpy.org/doc/stable/reference/generated/numpy.linalg.solve.html)
- 🤖 AI Prompt: "Give me a 2x2 and a 3x3 system of linear equations. Have me predict, just by looking at the coefficients, whether each system has a unique solution, no solution, or infinite solutions — then have me solve both by hand and check my prediction."

**This Week's Mini Exercise:**

### Week 4 — Determinants, Inverses & Rank

**Topics Covered:**
- The determinant — what it measures geometrically (how much a transformation scales area/volume), sign meaning (orientation flip)
- When a determinant is zero, and what that means about the transformation (collapses space into a lower dimension)
- The matrix inverse — what "undoing" a transformation means, and the condition for an inverse to exist
- Rank — the true dimensionality of a matrix's output space, and rank-deficient matrices

**Tools / Technologies Used:**
- Python 3.x, NumPy (`numpy.linalg.det`, `numpy.linalg.inv`, `numpy.linalg.matrix_rank`)

**Key Techniques / Concepts to Internalize:**
- The determinant being zero and the matrix having no inverse are the same fact viewed two ways — a transformation that collapses space (determinant zero) has literally destroyed information, so there's no way to undo it and recover the original input.
- Rank is the honest measure of "how much information does this matrix actually preserve" — a matrix can look large (many rows/columns) while having low rank, meaning much of that size is redundant. This exact idea reappears later as the motivation for dimensionality reduction (PCA, Week 6) and for low-rank fine-tuning techniques you'll meet in NLP103.
- Computing a matrix inverse by hand is worth doing once for genuine understanding, but in real code you almost never explicitly invert a matrix — `numpy.linalg.solve` is both faster and more numerically stable than `inv(A) @ b`. Knowing this now prevents a habit you'd otherwise have to unlearn.

**Practical Tips:**
- Common mistake: computing `np.linalg.inv(A) @ b` to solve `Ax = b` in real code — always prefer `np.linalg.solve(A, b)` and understand why (numerical stability, efficiency).
- You understood this week if you can explain, without computing anything, why a matrix with two identical rows must have a determinant of zero.
- Don't grind through large hand-computed determinants — 2×2 and 3×3 by hand is enough for intuition; let NumPy handle anything larger while you focus on interpreting the result.

**Recommended Resources:**
- 📺 YouTube: 3Blue1Brown — "Essence of Linear Algebra," Chapter 5 (The determinant) and Chapter 8 (Inverse matrices, rank)
- 📘 Book: *Mathematics for Machine Learning* — Chapter 2.4–2.6 (Vector spaces, rank, determinant)
- 📄 Docs: [numpy.org — `numpy.linalg` module overview](https://numpy.org/doc/stable/reference/routines.linalg.html)
- 🤖 AI Prompt: "Give me a 3x3 matrix that is rank-deficient (rank 2, not 3) but doesn't obviously look that way at first glance. Ask me to explain, using the rows or columns, why it's rank-deficient before I check with NumPy."

**This Week's Mini Exercise:**
Generate 5 random 3×3 matrices, compute the determinant and rank of each with NumPy, and for any with a determinant near zero, identify (by inspecting the matrix) which row or column is a near-linear-combination of the others.

### Week 5 — Eigenvalues, Eigenvectors & Diagonalization

**Topics Covered:**
- What an eigenvector is: a direction a transformation only scales, never rotates off its own line
- Eigenvalues as the scaling factor along an eigenvector's direction
- Computing eigenvalues/eigenvectors conceptually (characteristic equation) and practically (`numpy.linalg.eig`)
- Diagonalization — expressing a matrix in terms of its eigenvectors/eigenvalues, and why that representation is useful (e.g., computing `A^n` cheaply)

**Tools / Technologies Used:**
- Python 3.x, NumPy (`numpy.linalg.eig`), Matplotlib

**Key Techniques / Concepts to Internalize:**
- An eigenvector isn't a special mathematical curiosity — it's the answer to "in which directions does this transformation behave as simply as possible (pure scaling)?" This exact question is what PCA (Week 6) is built on: finding the directions of a dataset along which variance is purest.
- Eigenvalues and eigenvectors always come in pairs for a reason: the eigenvector defines the direction, the eigenvalue defines what happens to it — neither means anything without the other.
- Diagonalization matters practically because repeated application of a matrix (`A @ A @ A @ ...`) is trivial once you know the eigen-decomposition — this connects directly to why certain recurrent computations are analyzed via eigenvalues (e.g., understanding gradient explosion/vanishing in DL104's RNNs).

**Practical Tips:**
- Common mistake: memorizing the characteristic equation `det(A - λI) = 0` without connecting it back to "eigenvectors are the directions the transformation doesn't rotate" — the formula should feel like a derivation of that idea, not an arbitrary rule.
- You understood this week if, given a diagonal matrix, you can immediately state its eigenvalues and eigenvectors without computing anything (they're just the diagonal entries and the standard basis vectors).
- Not every matrix has real eigenvalues (some have complex ones, e.g. pure rotation matrices) — don't get stuck trying to force a geometric "scaling direction" interpretation onto every single matrix; know this is a real exception, not a mistake in your understanding.

**Recommended Resources:**
- 📺 YouTube: 3Blue1Brown — "Essence of Linear Algebra," Chapter 14 (Eigenvectors and eigenvalues)
- 📘 Book: *Mathematics for Machine Learning* — Chapter 4.2 (Eigenvalues and Eigenvectors)
- 📄 Docs: [numpy.org — `numpy.linalg.eig`](https://numpy.org/doc/stable/reference/generated/numpy.linalg.eig.html)
- 🤖 AI Prompt: "Give me a 2x2 matrix with two clean, easy-to-compute-by-hand real eigenvalues. Have me find the eigenvalues and eigenvectors by hand step by step, checking my work at each stage before I verify with NumPy."

**This Week's Mini Exercise:**
Take a 2×2 matrix, compute its eigenvectors with NumPy, then plot the matrix's effect on a circle of points (a unit circle transformed by the matrix becomes an ellipse) and overlay the eigenvectors on the plot — visually confirming they align with the ellipse's axes.

### Week 6 — Norms, Orthogonality & Dimensionality Reduction

**Topics Covered:**
- Orthogonal and orthonormal vectors and bases
- Projections onto subspaces
- Singular Value Decomposition (SVD) — what it decomposes a matrix into, and why it's more broadly applicable than eigendecomposition (works for non-square matrices)
- Principal Component Analysis (PCA) as a direct application: using eigenvectors of a covariance matrix (or SVD) to find the directions of maximum variance in real data

**Tools / Technologies Used:**
- Python 3.x, NumPy (`numpy.linalg.svd`), scikit-learn (`sklearn.decomposition.PCA`, for comparison only), Matplotlib

**Key Techniques / Concepts to Internalize:**
- SVD generalizes eigendecomposition to any matrix, not just square ones — this is why it's the more commonly used tool in real ML systems (dimensionality reduction, recommender systems, and later, understanding embedding matrices).
- PCA isn't a separate algorithm to memorize — it's "compute the covariance matrix of your data, find its eigenvectors, and those eigenvectors are the new axes that capture the most variance." Every piece of that sentence should already make sense from this course.
- This week is intentionally the bridge into ML101 — the goal isn't mastering PCA as an ML technique yet (that's ML101/ML102's job), it's recognizing that "dimensionality reduction" is a linear algebra idea wearing an ML name.

**Practical Tips:**
- Common mistake: treating PCA as a black-box scikit-learn function without connecting it to eigenvectors of a covariance matrix — if you can't explain what scikit-learn's `PCA` is doing internally after this week, revisit rather than move on.
- You understood this week if you can implement a basic 2-component PCA from scratch using only NumPy (covariance matrix + eigendecomposition) and get results that visually match scikit-learn's output on the same toy data.
- Don't chase every detail of SVD's derivation — the practical goal is knowing what it decomposes a matrix into (`A = UΣVᵀ`) and why that's useful, not re-deriving the algorithm that computes it.

**Recommended Resources:**
- 📺 YouTube: 3Blue1Brown — "Essence of Linear Algebra" wrap-up (dot products/projections, Chapter 9), plus StatQuest — "Principal Component Analysis (PCA), Step-by-Step"
- 📘 Book: *Mathematics for Machine Learning* — Chapter 10 (Dimensionality Reduction with PCA)
- 📄 Docs: [numpy.org — `numpy.linalg.svd`](https://numpy.org/doc/stable/reference/generated/numpy.linalg.svd.html)
- 🤖 AI Prompt: "Walk me through implementing PCA from scratch using only NumPy: computing a covariance matrix, finding its eigenvectors, and projecting data onto the top 2 principal components. Have me write each step and explain what it's doing before showing me if my code is correct."

**This Week's Mini Exercise:**

## Main Project

**Project: Image Compression via SVD**

A from-scratch (NumPy-only) tool that compresses a grayscale image using Singular Value Decomposition, demonstrating a direct, visual application of everything in this course.

**Requirements:**
- Load a grayscale image as a 2D NumPy array (a matrix of pixel intensities).
- Compute its SVD using `numpy.linalg.svd`.
- Reconstruct an approximation of the image using only the top-*k* singular values/vectors, for several values of *k* (e.g. 5, 20, 50, 100).
- Display or save the original image alongside each compressed reconstruction, with the compression ratio (original data size vs. reconstructed data size at that *k*) labeled on each.
- Include a short written explanation (in the project's README) of what the top singular values represent, in your own words, and why keeping only a few still produces a recognizable image.

**Stretch Goals:**
- Extend the tool to work on color images (apply SVD per color channel).
- Plot the singular values in descending order and use the plot to justify a "good" choice of *k* for this specific image (where the curve flattens out).
- Compare your from-scratch SVD compression visually and numerically (e.g., using mean squared error) against JPEG compression at a roughly equivalent file size.

**What "Done" Looks Like:**
- The tool runs end-to-end on at least one real image you provide, producing a clear before/after comparison.
- You can explain, without notes, why a low-rank approximation using only the top-*k* singular values still resembles the original image.
- The code is organized into functions (not one long script), pushed to this repo's `Projects/` folder with a README including the compressed output images.
