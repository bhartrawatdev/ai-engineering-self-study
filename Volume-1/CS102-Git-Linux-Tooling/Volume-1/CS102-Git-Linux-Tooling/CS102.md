# CS102 — Git, Linux & Dev Tooling

## Why This Matters

You can already write Python from CS101 — this course is about not fighting the environment around the code. Every real engineering job (and every later course in this program that touches a cloud instance, a training job, or a deployed model) assumes you're comfortable in a terminal and can use Git without pausing to think. Skipping this is the single most common reason self-taught programmers hit a wall the moment work leaves their local machine or involves more than one person.

## Prerequisites

- **CS101 (Programming Foundations)** — you should already be comfortable writing and running Python scripts; this course doesn't teach programming, it teaches the environment around it.

## Where This Is Used Later

- **CS103 (Data Structures & Algorithms)** and every course after it — assumes you're working from a terminal and committing work to Git without being told to.
- **AIE104 (MLOps & Model Deployment)** — builds directly on shell fluency and Git workflows for CI/CD, deployment scripts, and remote server work.
- **AIE105 (Scalable AI Systems & Infrastructure)** — cloud/remote instance work assumes SSH and shell comfort are already second nature.
- **The Main Project of every course from here on** — you'll be expected to push work to this repo using a real Git workflow, not just drag-and-drop files.

## Full Topic Breakdown

- **Linux / Shell Fundamentals**
  - Filesystem navigation: `pwd`, `cd`, `ls`, absolute vs. relative paths
  - File operations: `cp`, `mv`, `rm`, `mkdir`, `touch`
  - Viewing files: `cat`, `less`, `head`, `tail`
  - Permissions: `chmod`, `chown`, the read/write/execute model
  - Redirection and piping: `>`, `>>`, `<`, `|`
  - Process management: `ps`, `top`/`htop`, `kill`, running processes in the background (`&`, `nohup`)
  - Environment variables: `export`, `.bashrc`/`.zshrc`, `$PATH`
- **Text Processing & Shell Power Tools**
  - `grep` — searching text, regex flags
  - `find` — locating files by name, type, modification time
  - `sed` — basic stream editing (substitution)
  - `awk` — basic column-based text processing
  - Combining tools with pipes to build one-liners
  - Basic Bash scripting: variables, conditionals, loops, arguments
- **Git Fundamentals**
  - What version control actually solves (the problem before the tool)
  - `git init`, the working directory / staging area / repository model
  - `git add`, `git commit`, writing good commit messages
  - `git status`, `git log`, `git diff`
  - `.gitignore` and what should never be committed
  - Branching: `git branch`, `git checkout` / `git switch`
- **Git Collaboration & GitHub Workflow**
  - Remotes: `git remote`, `git push`, `git pull`, `git fetch`
  - Merging vs. rebasing — what each actually does to history
  - Resolving merge conflicts by hand
  - GitHub-specific workflow: forking, pull requests, code review basics, issues
  - Undoing things safely: `git reset`, `git revert`, `git checkout -- <file>`, reflog as a safety net
- **Dev Environment & Productivity Tooling**
  - SSH keys: generating, adding to GitHub, using for authentication
  - Dotfiles: what they are, why version-controlling your own config matters
  - VS Code configuration: settings sync, useful extensions, integrated terminal
  - Terminal multiplexers (`tmux`) — conceptual intro, not mastery
  - Putting it together: a terminal-first daily workflow

## Weekly Study Plan

### Week 1 — Linux & the Command Line Fundamentals

**Topics Covered:**
- Filesystem navigation: `pwd`, `cd`, `ls` (and useful flags: `-la`, `-lh`)
- Absolute vs. relative paths, `.` and `..`, `~`
- File operations: `cp`, `mv`, `rm`, `mkdir`, `touch`, `rmdir`
- Viewing file contents: `cat`, `less`, `head`, `tail` (including `tail -f`)
- Basic permissions: reading `ls -l` output, `chmod` in symbolic and numeric form

**Tools / Technologies Used:**
- A real terminal — WSL2 if on Windows, native terminal on macOS/Linux
- Bash or Zsh (whichever your system defaults to)

**Key Techniques / Concepts to Internalize:**
- The filesystem is a tree, and "where am I" (`pwd`) should be a reflex before running any command that touches files — most beginner Git/shell disasters come from running a destructive command in the wrong directory.
- Permissions aren't arbitrary — the read/write/execute model applied separately to owner/group/other is the same model you'll rely on later when deploying to remote servers.
- `rm` has no undo. Internalizing that now, with low stakes, is cheaper than learning it later with something that matters.

**Practical Tips:**
- Common mistake: using `rm -rf` without being certain of `pwd` first — always run `pwd` and `ls` before any destructive command until it's automatic to check.
- You understood this week if you can navigate to any file on your system, view it, copy it, and rename it without needing to look up syntax.
- If you're on Windows, get WSL2 set up properly this week rather than trying to learn "Windows-flavored" command line tools — everything downstream in this program assumes a Unix-like shell.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "Learn the Linux Command Line" (first ~45 minutes)
- 📘 Book: *The Linux Command Line* by William Shotts — Chapters 1–4 (free PDF available from the author)
- 📄 Docs: [ubuntu.com/tutorials — Command line for beginners](https://ubuntu.com/tutorials/command-line-for-beginners)
- 🤖 AI Prompt: "I'm new to the Linux command line. Give me a realistic scenario involving navigating folders, copying files, and checking permissions, then walk me through solving it command by command, explaining each flag."

**This Week's Mini Exercise:**
Starting from your home directory, create a folder structure three levels deep, populate it with a few text files using `touch`, move one file up a level and rename another, then delete the whole structure — all without leaving the terminal or looking anything up mid-task.

### Week 2 — Text Processing & Shell Power Tools

**Topics Covered:**
- Redirection and piping: `>`, `>>`, `<`, `|`
- `grep` — text search, common flags (`-i`, `-r`, `-n`, `-v`)
- `find` — locating files by name, type, and modification time
- `sed` — basic substitution (`s/old/new/`)
- `awk` — basic column extraction and simple field processing
- Combining tools into one-liners
- Basic Bash scripting: variables, `if`, `for`, reading arguments (`$1`, `$@`)

**Tools / Technologies Used:**
- Bash, `grep`, `find`, `sed`, `awk`

**Key Techniques / Concepts to Internalize:**
- Piping is the actual "Unix philosophy" in practice — small tools that each do one thing, chained together, are more powerful than one tool trying to do everything. This is the same design instinct that shows up later in good software architecture.
- `grep`/`sed`/`awk` overlap in what they *can* technically do, but each has a natural job: search vs. substitute vs. column processing. Reaching for the right one by default (rather than forcing everything through `awk`) is a sign of real fluency, not just memorized syntax.
- A Bash script is just a sequence of the same commands you already type manually — the mental shift is realizing scripting isn't a separate skill, it's automating what your hands already know.

**Practical Tips:**
- Common mistake: reaching for a full Python script to do something a one-line `grep`/`awk` pipe would solve in seconds — shell tools are usually faster for quick text/log processing tasks.
- You understood this week if, given a large log file, you can find all lines containing "ERROR", extract just the timestamps, and count them — as a single piped command.
- Don't try to memorize every `sed`/`awk` flag — build a small personal cheat-sheet of the 5–6 patterns you actually use, and look up the rest as needed. Real fluency here is decades of using it, not one week of study.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "Learn the Linux Command Line" (grep/sed/awk + piping sections)
- 📘 Book: *The Linux Command Line* by William Shotts — Chapters 7 (I/O Redirection) and 20 (Text Processing)
- 📄 Docs: [gnu.org/software/grep/manual](https://www.gnu.org/software/grep/manual/grep.html)
- 🤖 AI Prompt: "Give me a sample log file with a mix of INFO/WARNING/ERROR lines and timestamps. Ask me to write a single piped shell command to extract and count ERROR lines by hour, then check my answer and explain a cleaner way to do it if mine was overcomplicated."

**This Week's Mini Exercise:**
Using the synthetic log file you generated for the CS101 Main Project, write a single piped shell command (no Python) that counts how many `ERROR` lines occurred in each hour, sorted from most to least frequent.

### Week 3 — Git Fundamentals

**Topics Covered:**
- What version control actually solves — the "why," not just the commands
- `git init`, and the working directory / staging area / repository (three-tree) model
- `git add`, `git commit`, and what actually makes a good commit message
- `git status`, `git log` (including `--oneline` and `--graph`), `git diff`
- `.gitignore` — what should never be committed (secrets, `venv/`, `__pycache__/`, large data files)
- Branching basics: `git branch`, `git switch` / `git checkout -b`

**Tools / Technologies Used:**
- Git (CLI), a GitHub account (created this week if you don't have one)

**Key Techniques / Concepts to Internalize:**
- The three-tree model (working directory → staging area → repository) is the actual mental model that makes `git add` before `git commit` make sense — it's not a needless extra step, it's a deliberate "choose exactly what goes in this commit" mechanism.
- A commit is a snapshot with a message that should answer "why," not "what" — `git diff` already shows *what* changed; a good message explains *why* it changed.
- Branches are cheap, disposable pointers, not heavyweight copies — the fear of "messing up the branch" usually comes from not yet trusting that you can always make a new one.

**Practical Tips:**
- Common mistake: writing commit messages like "fixed stuff" or "updates" — force yourself to write one clear sentence describing the actual change, even for tiny commits.
- You understood this week if you can explain, without looking it up, what the difference is between a file that's "modified," "staged," and "committed."
- Commit small and often this week even though it feels excessive — the habit of small, focused commits is much easier to build now than to retrofit later.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "Git and GitHub for Beginners" (first half, through committing and branching)
- 📘 Book: *Pro Git* by Scott Chacon & Ben Straub — Chapter 1–3 (free online)
- 📄 Docs: [git-scm.com/book/en/v2](https://git-scm.com/book/en/v2)
- 🤖 AI Prompt: "I'm new to Git. Explain the working directory, staging area, and repository using a physical, non-computer analogy. Then give me a short sequence of file changes and ask me to predict what `git status` would show at each step."

**This Week's Mini Exercise:**
Turn last week's Bash one-liner exercise into a small Git repo: `git init`, add a proper `.gitignore`, and make at least 5 separate, well-messaged commits building up a short README that documents the commands you learned — no single giant commit.

### Week 4 — Git Collaboration & GitHub Workflow

**Topics Covered:**
- Remotes: `git remote add`, `git push`, `git pull`, `git fetch`
- Merging vs. rebasing — what each does to commit history, and when to use which
- Resolving a merge conflict by hand
- GitHub workflow: forking vs. branching, pull requests, basic code review etiquette, issues
- Undoing things safely: `git reset` (soft vs. hard), `git revert`, `git checkout -- <file>`, and the reflog as a last-resort safety net

**Tools / Technologies Used:**
- Git (CLI), GitHub (web + CLI if you want `gh`)

**Key Techniques / Concepts to Internalize:**
- Merge and rebase solve the same problem (combining diverging history) with different tradeoffs — merge preserves exact history including the divergence, rebase rewrites history into a straight line. Neither is "correct"; knowing *why* you'd choose one matters more than memorizing the commands.
- A merge conflict is not a failure state — it's Git correctly refusing to guess when two changes touched the same lines. Resolving one by hand, deliberately, removes the fear response most beginners have to seeing "CONFLICT" in the terminal.
- `git reset --hard` can lose work; `git revert` cannot (it adds a new commit undoing changes instead of rewriting history) — knowing which "undo" tool is destructive and which is safe is the actual skill, not just knowing both exist.

**Practical Tips:**
- Common mistake: panicking at a merge conflict and running `git checkout .` or similar to "make it go away," which can silently discard real work — read what Git is actually telling you first.
- You understood this week if you can deliberately create a merge conflict between two branches, resolve it by hand, and explain what you chose and why.
- Practice a pull request end-to-end this week even solo — open one against your own repo, from a feature branch, and merge it through the GitHub UI, not just the CLI. The PR workflow itself is the point, not just the Git mechanics underneath it.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp — "Git and GitHub for Beginners" (second half — remotes, merge conflicts, PRs)
- 📘 Book: *Pro Git* — Chapter 3.2 (Basic Branching and Merging) and Chapter 6.1 (GitHub basics)
- 📄 Docs: [docs.github.com/en/pull-requests](https://docs.github.com/en/pull-requests)
- 🤖 AI Prompt: "Walk me through a scenario where two branches both modified the same line of the same file. Have me simulate resolving the conflict step by step, and after I describe my resolution, tell me if my understanding of merge vs. rebase in that scenario is correct."

**This Week's Mini Exercise:**
Push your Week 3 repo to GitHub, create a feature branch, make a change that deliberately conflicts with a change on `main`, merge `main` into your branch to trigger the conflict, resolve it by hand, then open and merge a pull request through the GitHub UI.

### Week 5 — Dev Environment & Productivity Tooling

**Topics Covered:**
- SSH keys: generating (`ssh-keygen`), adding to GitHub, understanding public/private key auth conceptually
- Dotfiles: what they are (`.bashrc`/`.zshrc`, Git config, editor config) and why version-controlling your own is worth doing
- VS Code configuration: settings, useful extensions for Python/Git work, the integrated terminal
- `tmux` — a conceptual first look (sessions, panes, detach/reattach) — not mastery
- Environment variables in practice: `export`, `.env` files, why secrets don't belong in code or Git history

**Tools / Technologies Used:**
- SSH, Git, VS Code, `tmux` (optional but recommended), a dotfiles repo on GitHub

**Key Techniques / Concepts to Internalize:**
- SSH key auth isn't just "a more convenient login" — understanding the public/private key model conceptually (you keep the private key, the server only ever needs the public one) is the same model behind API key security you'll deal with constantly from Volume 5 onward.
- A dotfiles repo turns "how I have my environment set up" from tribal knowledge stuck on one machine into something reproducible — this becomes genuinely useful the first time you set up a new machine or a remote instance.
- Secrets (API keys, tokens, passwords) belong in environment variables or `.env` files that are explicitly gitignored — never hardcoded in a script that might get committed. This habit matters far more once you're working with paid API keys in Volume 5–6.

**Practical Tips:**
- Common mistake: committing a `.env` file or an API key directly into a script "just for now" — set up the gitignore habit before you ever have a real key to protect, not after you accidentally leak one.
- You understood this week if you can set up SSH auth to GitHub from a fresh terminal without needing to look anything up, and explain in one sentence why it's more secure than password auth.
- Don't over-invest in `tmux` mastery yet — a basic session/pane understanding is enough for now; real fluency builds naturally once you're doing remote server work in Volume 6.

**Recommended Resources:**
- 📺 YouTube: freeCodeCamp or ThePrimeagen — any solid "dotfiles and terminal setup" walkthrough (search current, since setups get updated often)
- 📘 Book: *The Missing Semester of Your CS Education* (MIT, free online) — the "Command-line Environment" and "Security and Cryptography" lectures
- 📄 Docs: [docs.github.com/en/authentication/connecting-to-github-with-ssh](https://docs.github.com/en/authentication/connecting-to-github-with-ssh)
- 🤖 AI Prompt: "Explain how SSH public/private key authentication works using a real-world analogy involving a lockbox, not the usual generic explanation. Then give me a short checklist to verify my own SSH setup with GitHub is actually working correctly."

**This Week's Mini Exercise:**
Generate a new SSH key, add it to your GitHub account, and clone/push to a repo using SSH instead of HTTPS. Then start a `dotfiles` repo containing at least your Git config and shell aliases, with a `.gitignore` that would prevent an `.env` file from ever being committed by accident.

## Main Project

**Project: Personal Dev Environment + End-to-End Git Workflow**

Two connected deliverables that together prove you can operate like an engineer, not just write code.

**Requirements:**

*Part A — Dotfiles Repo:*
- A public GitHub repo containing your shell config (`.bashrc`/`.zshrc` or equivalent), Git config (`.gitconfig`), and any editor settings you want version-controlled.
- A `.gitignore` that explicitly excludes secrets/`.env` files, with a comment explaining why.
- A `README.md` explaining what's in the repo and how to apply it to a fresh machine (even a simple manual copy-steps README is fine — a setup script is a stretch goal, not a requirement).

*Part B — Simulated Collaboration Workflow:*
- Using any small script (the CS101 Log Analyzer works well), simulate a realistic feature-branch workflow on GitHub:
  - At least 3 feature branches, each for one distinct change
  - At least one deliberate merge conflict, resolved by hand
  - At least one pull request opened and merged through the GitHub UI, with a real description of the change
  - Clean, well-messaged commit history — no "fix," "wip," or "asdf" commits

**Stretch Goals:**
- Write a simple shell script that automates applying your dotfiles to a fresh environment (symlinking configs into place).
- Set up and use `tmux` for the collaboration-workflow part of this project instead of a single terminal window.
- Practice the same feature-branch workflow using `rebase` instead of `merge` for at least one branch, and note the difference in the resulting history.

**What "Done" Looks Like:**
- Your dotfiles repo is real — it reflects your actual current setup, not a fabricated example.
- You can walk through `git log --oneline --graph` on the collaboration-workflow repo and narrate the story of what happened at each point without hesitation.
- SSH auth to GitHub works from your primary machine.
- Both repos are linked from this course folder's `Projects/` file.
