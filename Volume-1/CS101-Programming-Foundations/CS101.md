# CS101 — Programming Foundations

## Why This Matters

Every course after this one assumes you can write, read, and debug Python without thinking about the syntax. If that's not automatic yet, you'll be fighting the language instead of learning the actual concept in every later course — from NumPy in MA101 to PyTorch in DL102. This course exists to make Python fluent enough that it disappears into the background.

## Prerequisites

None. This is the entry point of the program. If you've never written code before, start here at Week 1. If you already know basic syntax but haven't worked with error handling, file I/O, or writing real multi-file scripts, skim Weeks 1–3 and slow down from Week 4 on.

## Where This Is Used Later

- **CS102 (Git, Linux & Dev Tooling)** — assumes you can already write and run scripts; focuses purely on the environment around the code.
- **CS103 (Data Structures & Algorithms)** — assumes fluency with Python's built-in data structures (lists, dicts, sets) so it can focus on complexity and custom structures instead of syntax.
- **MA101 (Linear Algebra for ML)** — the NumPy work there assumes comfortable Python, not first-time programming.
- **Every course from Volume 3 onward** — all ML/DL work is written in Python; this is the floor everything stands on.

## Full Topic Breakdown

- **Core Syntax & Execution**
  - Variables, dynamic typing, naming conventions
  - Primitive types: `int`, `float`, `str`, `bool`, `None`
  - Operators: arithmetic, comparison, logical, assignment (including augmented)
  - Running scripts vs. using the REPL vs. Jupyter notebooks
- **Control Flow**
  - Conditionals: `if` / `elif` / `else`, truthy/falsy values
  - Loops: `for`, `while`, `break`, `continue`, `else` on loops
  - Comprehensions: list, dict, set, generator
- **Core Data Structures**
  - Lists — indexing, slicing, mutation, common methods
  - Tuples — immutability, unpacking
  - Dictionaries — keys/values, iteration, `.get()`, nested dicts
  - Sets — uniqueness, set operations (union, intersection, difference)
  - Choosing the right structure for a problem
- **Functions**
  - Definition, parameters, return values
  - Default arguments, `*args`, `**kwargs`
  - Scope: local vs. global, the LEGB rule
  - Recursion
  - Functions as first-class objects: passing functions, lambdas, `map`/`filter`/`sorted(key=...)`
  - Closures (conceptual introduction)
- **Strings & Text Processing**
  - String methods, slicing, formatting (f-strings)
  - Regular expressions (basics: `re.search`, `re.match`, `re.findall`)
- **File I/O**
  - Reading/writing text files, `with` statements and context managers
  - Working with CSV and JSON via the standard library
- **Error Handling**
  - `try` / `except` / `else` / `finally`
  - Raising exceptions, custom exception classes
  - Why silent failure is worse than a crash
- **Modules & the Standard Library**
  - Importing, `import` vs `from...import`
  - Writing your own modules, `__main__` guard
  - Useful standard library modules: `os`, `sys`, `pathlib`, `datetime`, `itertools`, `collections`, `argparse`
  - Installing and using third-party packages with `pip`
- **Light Introduction to OOP**
  - Classes, `__init__`, instance attributes and methods
  - Just enough to read and write simple classes — full OOP depth is CS104's job, not this course's
- **Working With Data (bridge to later ML work)**
  - Intro to NumPy: arrays, basic vectorized operations
  - Why vectorization matters (a first taste, not a deep dive — that's MA101/DL101)

### Week 2 — Control Flow

**Topics Covered:**
- `if` / `elif` / `else` chains
- `for` loops over ranges, strings, and lists
- `while` loops and when to prefer them over `for`
- `break`, `continue`, and the lesser-known `else` clause on loops
- List, dict, and set comprehensions (introductory level)

**Tools / Technologies Used:**
- Python 3.x, VS Code

**Key Techniques / Concepts to Internalize:**
- Comprehensions aren't just shorter syntax — they express "build a new collection from this one" as a single readable expression, which matters for how you'll think about data transformations later (this is the same mental model as pandas/NumPy operations down the line).
- Knowing *when not* to use a comprehension — if it needs a comment to explain it, it should probably be a loop.
- The loop `else` clause runs when a loop completes without hitting `break` — rare but useful for search patterns.

**Practical Tips:**
- Common mistake: off-by-one errors with `range()` — `range(5)` is `0` to `4`, not `1` to `5`.
- You understood this week if you can rewrite any basic `for` loop that builds a list as a list comprehension, and explain in one sentence why the comprehension is or isn't more readable.
- Spend at least a full day just on comprehensions — they show up everywhere in real Python code and are a common self-taught blind spot.

### Week 3 — Core Data Structures

**Topics Covered:**
- Lists: indexing, negative indexing, slicing, mutation, common methods (`append`, `extend`, `sort`, `pop`, etc.)
- Tuples: immutability, unpacking, when to use over lists
- Dictionaries: creation, key/value iteration, `.get()`, `.items()`, nested dictionaries
- Sets: uniqueness guarantees, union/intersection/difference
- Choosing the right structure for a given problem

**Tools / Technologies Used:**
- Python 3.x, VS Code

**Key Techniques / Concepts to Internalize:**
- Mutability vs. immutability isn't academic — it directly explains bugs like "why did modifying my copy change the original list" (shared references).
- Dictionaries as the general-purpose "lookup" tool — recognizing when a problem is secretly a dictionary problem (e.g., counting occurrences, grouping).
- Sets exist for a reason beyond "list without duplicates" — fast membership testing (`in`) is O(1) on a set vs. O(n) on a list, which matters as data grows.

**Practical Tips:**
- Common mistake: `list_b = list_a` doesn't copy — it's the same list under two names. Use `list_a.copy()` or `list(list_a)` for an actual copy.
- You understood this week if, given a word-counting problem, you reach for a dictionary immediately instead of two parallel lists.
- Slicing (`list[1:3]`, `list[::-1]`) deserves deliberate practice — it's used constantly and easy to get backwards under pressure.

### Week 4 — Functions Deep Dive & Scope

**Topics Covered:**
- Function definitions, parameters, return values
- Default arguments, `*args`, `**kwargs`
- Scope: local vs. global, the LEGB rule
- Recursion, including base cases and stack depth intuition
- Functions as first-class objects: passing functions as arguments, `lambda`, `map`/`filter`/`sorted(key=...)`
- Closures — a conceptual first look, not mastery

**Tools / Technologies Used:**
- Python 3.x, VS Code

**Key Techniques / Concepts to Internalize:**
- A function is just a value like any other in Python — this is the idea that makes `sorted(data, key=lambda x: x[1])` make sense instead of feeling like magic syntax.
- Recursion isn't "a different kind of loop" — it's a different way of thinking about a problem as smaller versions of itself; the mental shift matters more than the syntax.
- Mutable default arguments (`def f(x, items=[])`) are a classic Python trap — understanding *why* it's a trap teaches you something real about how Python evaluates default arguments (once, at definition time).

**Practical Tips:**
- Common mistake: confusing `*args` (positional, becomes a tuple) with `**kwargs` (keyword, becomes a dict) — say them out loud as "star-args, double-star-kwargs" until the distinction is automatic.
- You understood this week if you can explain what a closure is without using the word "closure" — describe it as "a function that remembers the environment it was created in."
- Spend real time on recursion even if loops feel easier — tree/graph traversal in CS103 and backprop's computational graph later both build on this way of thinking.

**Recommended Resources:**
- 📺 YouTube: Corey Schafer — "Python OOP Tutorials" playlist intro on functions, and "*args and **kwargs"
- 📘 Book: *Python Crash Course* — Chapter 8
- 📄 Docs: [docs.python.org/3/tutorial/controlflow.html#defining-functions](https://docs.python.org/3/tutorial/controlflow.html#defining-functions)
- 🤖 AI Prompt: "Explain Python closures using a real-world analogy (not a banking/counter example — pick something unusual). Then give me a broken closure example with a common bug and ask me to spot and fix it."

**This Week's Mini Exercise:**

### Week 5 — Strings, File I/O & Error Handling

**Topics Covered:**
- String methods, slicing, f-string formatting (including format specs)
- Basic regular expressions: `re.search`, `re.match`, `re.findall`
- Reading/writing text files with `with` statements
- Working with CSV and JSON via `csv` and `json` modules
- `try` / `except` / `else` / `finally`
- Raising exceptions, writing custom exception classes

**Tools / Technologies Used:**
- Python 3.x, `re`, `csv`, `json` standard library modules, VS Code

**Key Techniques / Concepts to Internalize:**
- `with` statements aren't just "nicer syntax for files" — they guarantee cleanup (closing the file) even if an error happens mid-block. This is the context manager pattern you'll see again later (e.g., in PyTorch's `no_grad()`).
- Catching an exception should be a deliberate decision about what you can actually recover from — catching `Exception` broadly to "make the error go away" hides real bugs.
- Regex is a tool for *pattern* matching, not general parsing — knowing when reaching for regex is the wrong call (e.g., don't regex-parse JSON) is as important as knowing the syntax.

**Practical Tips:**
- Common mistake: opening a file without `with` and forgetting to close it, or catching exceptions so broadly that a typo in your own code gets silently swallowed.
- You understood this week if you can explain the difference between `except Exception:` and `except FileNotFoundError:` and articulate why the specific one is almost always better.
- Don't over-invest in regex mastery yet — get comfortable with the basics (`findall`, groups) and move on; deep regex fluency comes with repetition over time, not a single week.

**Recommended Resources:**
- 📺 YouTube: Corey Schafer — "Python Exception Handling" and "Working with JSON Data"
- 📘 Book: *Python Crash Course* — Chapter 10 (Files and Exceptions)
- 📄 Docs: [docs.python.org/3/tutorial/errors.html](https://docs.python.org/3/tutorial/errors.html), [docs.python.org/3/library/json.html](https://docs.python.org/3/library/json.html)

### Week 7 — Light Introduction to OOP

**Topics Covered:**
- Classes and objects: `__init__`, instance attributes, instance methods
- The difference between a class and an instance
- `self`, and what it actually refers to
- A first look at `__str__`/`__repr__` for readable object printing
- Just enough to read and write simple classes — inheritance, polymorphism, and design patterns are CS104's job

**Tools / Technologies Used:**
- Python 3.x, VS Code

**Key Techniques / Concepts to Internalize:**
- A class is a blueprint, an object is a specific thing built from it — sounds obvious stated plainly, but the confusion between "the class" and "an instance of the class" is the most common early OOP stumbling block.
- `self` isn't a keyword — it's just the conventional name for "the instance calling this method," and understanding that Python passes it automatically demystifies a lot of confusing method signatures.
- This week is intentionally shallow. Going deep into inheritance/polymorphism here would front-load complexity that CS104 covers properly — resist the urge to over-build.

**Practical Tips:**
- Common mistake: forgetting `self` as the first parameter in instance methods, or confusing class-level attributes with instance-level ones.
- You understood this week if you can explain, in your own words, why you'd use a class instead of just a dictionary for the same data — hint: it's about bundling data with the behavior that operates on it.
- Don't chase every OOP concept you find in a tutorial (metaclasses, abstract base classes, etc.) — that's explicitly out of scope until CS104.

**Topics Covered:**

- Vectorized operations vs. Python loops — a first taste of *why* this matters for performance

**Tools / Technologies Used:**
- Python 3.x, NumPy, VS Code

**Key Techniques / Concepts to Internalize:**
- Vectorization isn't just "NumPy is faster" — it's a different way of expressing "do this to every element" that avoids explicit Python-level loops, and this exact mental model reappears constantly in MA101 and every DL framework after it.

**Practical Tips:**
- Common mistake: writing a NumPy array like a Python list and looping over it element-by-element — if you're writing a `for` loop over a NumPy array's elements, there's almost always a vectorized way to do it instead.
- You understood this week if you can look at a small function using explicit loops over lists and roughly sketch what the NumPy vectorized version would look like, even if you don't get the syntax perfect.
- This is a lighter week by design — it's meant to bridge into MA101, not exhaust the topic.

## Main Project

**Project: Command-Line Log Analyzer**

A CLI tool that reads a log file (you'll generate a synthetic one — format: `timestamp | level | message`, e.g. `2026-01-15 08:32:11 | ERROR | Connection timeout`) and produces a summary report.

**Requirements:**
- Accept the log file path as a command-line argument via `argparse`.
- Parse each line, handling malformed lines gracefully (log and skip, don't crash).
- Report: total line count, count by log level (`INFO`/`WARNING`/`ERROR`/etc.), the 5 most common error messages, and the busiest hour of the day (most log entries).
- Support an optional `--output` flag to write the report as JSON instead of printing to the console.
- Represent each parsed log entry as a small class (from Week 7), not a raw dictionary or tuple.
- Include at least one custom exception (e.g., `MalformedLogEntryError`) that you raise and catch deliberately.

**Stretch Goals:**
- Add a `--level` filter flag to only report on a specific log level.
- Use `regex` to support parsing a second, differently-formatted log file (e.g., a common real-world format like nginx access logs) with the same tool.
- Use NumPy to compute basic statistics (e.g., average log entries per hour) instead of hand-rolled loops.

**What "Done" Looks Like:**
- Running `python log_analyzer.py sample.log` produces a readable console report with no crashes on malformed input.
- The code is split across at least two files (e.g., `log_analyzer.py` and a `models.py` for the log entry class), imported properly as a module — not one giant script.
- You can explain every line of your own code without hesitation if asked.
- It's pushed to this repo's course folder under `Projects/`, with a short `README.md` inside describing how to run it.
